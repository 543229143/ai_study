## 个人定位

**Q：先介绍一下你自己。**
**A：** 我有二十年 Java 后端开发经验，之前主要负责互联网系统开发和微服务架构，最近几年转向 AI 应用开发，重点包括 RAG 知识库、智能体、大模型接口开发，还有企业 AI 应用落地。

**Q：现在学 AI 会不会有点晚？**
**A：** 说实话，如果只学几个 API 调用，确实没什么竞争力。但企业项目最终需要的不是大模型本身，而是系统设计、业务理解、数据治理、工程化落地，这些都需要非常丰富的经验，这恰恰是我的优势。

## 模型应用开发

**Q：大模型应用开发具体做什么？**
**A：** 我理解的大模型应用开发不是训练模型，而是利用大模型的能力去构建企业应用，比如智能客服、知识库助手、业务 Agent 这些。核心工作包括模型接入、提示词设计、RAG、工具调用、系统架构设计、效果评估等等。

**Q：和传统 Java 开发最大的区别在哪？**
**A：** 最大的区别就是不确定性。传统系统的输入输出是比较固定的，但大模型的输出有概率性，所以你没法保证它每次都说对。这就需要在系统层面加各种机制，比如检索、约束、评估，来让 AI 系统更可靠。

## RAG 基础

**Q：RAG 的工作原理是什么？**
**A：** RAG 就是检索增强生成，主要解决大模型的两个问题。一个是知识更新问题，企业内部的各种文档、规章制度、业务资料，大模型本身是不知道的；另一个是幻觉问题，模型不知道答案的时候，它可能会根据概率去生成一些看起来合理但实际上是错的答案。

整体的思路是这样的：用户提出问题之后，系统先把问题做向量化，转成向量，然后到向量数据库里去检索相关知识，找到相关的文档之后，把这些内容和问题一起发给大模型，模型基于企业真实的资料来生成答案。完整的链路就是文档采集、文档解析、切片向量化、入库，然后用户检索、组装 prompt、模型生成。

**Q：为什么不能把企业资料全部放到 prompt 里？**
**A：** 主要三个原因。第一，上下文窗口不是无限的，你不可能把所有东西都塞进去；第二，成本问题，你发大量无关的内容，token 消耗就上去了；第三，准确性问题，资料太多了反而会干扰模型判断，关键信息容易被淹没。所以企业应用一般不是让模型记住所有东西，而是让模型需要的时候能检索到正确的资料。

**Q：怎么区分普通问答和需要 RAG 检索的问答？**
**A：** 做了前置问题分类。日常通识问题、闲聊对话、基础常识，直接调大模型回答；涉及公司业务流程、规章制度、内部规范的问题，自动触发 RAG 检索，保证答案有据可依。

**Q：向量化是什么？**
**A：** 简单说就是把文本转成计算机能计算的数字向量。举个例子，"如何申请退款"和"订单取消怎么退款"，文字不一样但语义是相近的，向量化模型会把它们转成距离比较近的向量。这样系统检索的时候就不只是匹配关键词了，而是理解语义。

**Q：为什么不用 MySQL 的 LIKE 查询？**
**A：** LIKE 查询适合精确的关键词匹配，但用户的表达方式太多了。比如用户问"钱什么时候能退回来"，资料里写的是"退款周期一般是三到七个工作日"，关键词完全不沾边。所以生产环境一般会用关键词加上向量检索，再加一个重排序模型，这样召回的准确率才够。

## RAG 高级架构

**Q：多路召回加重排的方案，怎么理解？**
**A：** 本质上就是一个"粗筛加精选"的漏斗模型。多路召回负责把漏斗的口子开大，保证相关的文档都能被捞回来，宁可错杀一千不放过一个；重排负责在漏斗底部做最后的把关，用高精度的模型把最相关的 Top5 筛选出来喂给大模型，从源头上减少幻觉。

**Q：为什么不直接用向量模型做重排？**
**A：** 向量模型是 Bi-encoder，速度快，但匹配精度有限。重排用的 Cross-encoder 是把问题和文档拼在一起做全量交互的，精度高很多，能精准判断两者是不是真的相关。所以它们的定位不一样——Embedding 做初筛，Rerank 做精排。

**Q：两路检索各自适合什么场景？**
**A：** 向量检索适合语义比较模糊、需要理解上下文的场景，比如用户说"怎么申请退款"，它能召回语义相关的退款流程。BM25 这种传统检索适合精确关键词匹配，比如订单号、产品型号、身份证号这些需要完全命中的内容，向量检索很容易漏掉。

**Q：检索出来的内容很多，但回答反而变差了，怎么解决？**
**A：** 会的，核心原因是召回了大量弱相关、冗余的文档片段，干扰了模型判断。主要做两点优化：一是接 Rerank 重排序模型过滤无效内容，只保留 Top3 到 Top5 高关联片段；二是做切片内容去重，剔除相似冗余的内容，保证传给模型的信息精准有效。

**Q：召回不准的常见原因有哪些？**
**A：** 主要有四点。第一，文档切片不合理，内容碎片化或者冗余太多；第二，嵌入模型和当前业务不匹配，语义理解有偏差；第三，用户提问太口语化、表述模糊；第四，缺少重排序环节，大量弱相关内容干扰了精准召回。

**Q：RRF 算法是做什么的？**
**A：** RRF 就是倒数秩融合，它的核心作用是把两路检索的结果做一个平衡，不会让某一路的结果完全主导最终的排序，这样语义匹配和精准匹配的优势都能保留下来，比简单地把结果拼在一起效果好很多。

**Q：重排模型的精度和速度怎么平衡？**
**A：** 生产环境里一般是先靠多路召回把结果压缩到 100 篇以内，然后再用重排模型去处理，这样精度有保障，速度也不会太慢。如果并发很高的话，还可以给重排加缓存，重复的问题就不用重复打分。

**Q：这套方案能解决幻觉问题吗？**
**A：** 能大幅降低。因为多路召回保证了相关的资料都能被找到，重排又把最相关的内容筛出来了，大模型只能基于真实、精准的资料去回答，就不会凭空编造了，从根源上减少了幻觉。

## 切块与工程实践

**Q：文档切片大小一般怎么设？太短太长分别有什么问题？**
**A：** 日常业务文档我一般按 300 到 800 字符切，再配上 10% 到 20% 的重叠。切片太短的话，完整的业务逻辑会被拆散，模型拿到碎片化的信息理解不了上下文；切片太长呢，单条 token 占用高，冗余信息多，成本也高，还容易带进去无关内容干扰模型。重叠切片主要是为了防止关键描述刚好被切在两段中间，导致信息丢失。

**Q：嵌入模型怎么选？**
**A：** 通用业务用通用的中文嵌入模型就行，性价比高、响应也快。如果是金融、政务这种专业领域，用领域专项的嵌入模型，语义匹配精度会更高。生产环境的话，优先选稳定性好、向量维度和你的向量库能适配的模型。

**Q：重叠切片越多越好吗？**
**A：** 不是。重叠太小容易丢关键信息，重叠太大呢，切片冗余严重，入库数据量大，检索成本也高。我生产环境一般控制在 10% 到 20%，平衡准确率和性能。

## 降低幻觉

**Q：除了 RAG，还有哪些降低幻觉的手段？**
**A：** 有几个层面。第一，在 prompt 里强制约束，只能参考检索到的文档，没有对应资料就直接说不知道，禁止编造；第二，增加输出校验，把 AI 回答里的关键信息抽出来，反向去知识库核对一下准确性；第三，调低模型的 temperature，降低生成的随机性；第四，前面接一个小分类模型，提前识别那些知识库外的问题，直接拦住，不用走大模型。

## 向量数据库选型

**Q：向量数据库各自适合什么场景？**
**A：** 轻量化本地测试的话，用 Chroma、FAISS 就行，不用额外部署服务。中小型企业单机生产环境，用 Milvus 轻量版或者 Qdrant，部署简单也好维护。如果是海量文档、高并发分布式的场景，就用 Milvus 集群，支持分片扩容和数据持久化，适合正式生产。

## 性能优化

**Q：线上 RAG 并发高的时候有什么瓶颈？怎么优化？**
**A：** 主要三个瓶颈。一个是向量检索查询耗时比较长，一个是模型接口调用容易排队阻塞，还有一个是重复提问会造成重复的向量化计算，浪费资源。

优化的话，热点问题的检索结果可以做本地缓存，不用每次都查向量库；给向量库建索引加速相似度检索；文档向量化做成异步处理，不要同步阻塞；模型接口加队列限流，同时高频问答做缓存。

另外还有一个很多人容易忽略的点——成本问题。很多人只看 token 单价，但线上真正的瓶颈往往是 KV Cache 占显存，上下文越长并发越高，显存消耗越厉害，首 token 延迟也跟着飙升。

## 微服务 + AI

**Q：微服务经验怎么用在 AI 系统上？**
**A：** 我把 AI 系统拆成了多个独立的微服务，文档解析服务、向量检索服务、大模型网关服务、问答业务服务，每个职责清晰，完全解耦。沿用微服务那套成熟的方案，服务注册发现、熔断限流、日志链路追踪都能直接用。单独封装一个大模型统一网关，集中管理密钥、计费、鉴权，业务服务不用直接对接各家模型厂商的 API，后面换模型也很方便。

## Prompt 工程

**Q：Prompt 工程做过哪些优化？**
**A：** 几个手段。第一，分层设计，系统角色指令、知识库素材、用户提问分开，结构清晰；第二，输出格式约束，要求模型按固定 JSON 或者分段格式返回，方便后端解析；第三，Few-shot 示例，在 prompt 里加几个标准问答的例子，统一输出逻辑；第四，对内的和对外的 prompt 分开，精准控制回答的严谨度和话术风格。

另外还要注意区分 prompt engineering 和 context engineering——prompt 管的是"怎么说"，context 管的是"让模型在这一轮看到什么"。两件事是配合着做的，不是替代关系。

---

## 检索排序

**Q：检索出来多条文档，怎么判断优先级？**
**A：** 我做了三层排序。第一层，向量相似度粗召回，把匹配度靠前的先筛出来；第二层，用 Rerank 重排序模型精准判断关联度；第三层，加自定义权重规则，企业核心制度和最新的文档优先级高，老旧的降权，最后截取优质的 TopN 传给模型。

---

## 文档解析

**Q：企业内部文档格式很杂，怎么统一处理？**
**A：** 常规的 Word、PDF 用 PDFBox、POI 提取纯文本。扫描件和图片类的 PDF，接 OCR 识别。所有文档解析完了统一清洗，去掉页眉页脚、空行、乱码、水印这些无效内容，然后统一切片、向量化、入库。

## 传统系统接入 AI

**Q：传统 Java 系统接 AI，难点在哪？**
**A：** 传统系统都是确定性逻辑，AI 是概率性输出，接进来之后要兼容各种异常，比如模型调用超时、检索没结果、回答不规范等等。同时要做好数据隔离，防止企业私密资料外流，还要适配原有的权限体系，不同角色的员工只能检索自己权限范围内的文档。

**Q：只调 API 和真正做 AI 应用开发，差距在哪？**
**A：** 只调 API 只能做简单的演示问答，落不了地。真正的 AI 工程化开发，要打通文档处理、向量检索、数据治理、限流容错、成本管控、权限安全、效果迭代这一整条链路，这套工程化能力恰恰是资深后端开发者的核心竞争力。

---

## 权限安全

**Q：怎么防止用户越权查询？**
**A：** 入库和检索两层都做了权限管控。文档入库的时候，绑定对应的部门、角色、权限标签；用户检索的时候，携带他的权限信息去过滤结果，只能召回他权限范围内的资料，这样数据就隔离了，防止越权。

---

## 多轮对话

**Q：怎么做上下文多轮对话？**
**A：** 我对历史对话做了摘要压缩。用户第二轮提问的时候，系统结合上文的语义，重构出完整的问题再去做检索，这样就不会断章取义，多轮问答的连贯性就好很多。

---

## 异常处理

**Q：模型调用超时、报错、重试怎么处理？**
**A：** 按后端工程化的思路来处理。配置合理的超时时间、自动失败重试，同时加上熔断降级。模型调用异常的时候，给用户一个友好的提示，同时完整记录日志和异常链路，系统不能崩，用户体验不能受影响。

## RAG vs 微调

**Q：RAG 和微调怎么理解？怎么选？**
**A：** RAG 是外挂实时知识库，优势很明显——数据可以实时更新，不用改模型权重，没有训练成本，答案还能溯源，适合绝大多数企业场景。微调是改模型本身的权重参数，适合固定话术、固定风格、高频标准化的场景。企业落地优先用 RAG，灵活而且成本低；只有需要统一输出风格的时候，才用微调来辅助。

## Agent 系统设计

**Q：从零设计一个 Agent 系统，怎么做？**
**A：** 按七步走。第一步，定边界，哪些用 workflow 哪些用 agent；第二步，选模型，根据任务类型、延迟、成本来匹配；第三步，设计工具，最小权限原则，工具名要清楚、参数要结构化、约束尽量做到 schema 里；第四步，搭 Runtime，状态机加上工具执行器还有异常处理；第五步，加治理，人工介入、审计日志、权限控制，HITL 不是简单弹个框，是引入了一个 WAITING_HUMAN 状态，系统要能挂起和恢复；第六步，配记忆，临时状态、长期记忆、用户偏好分开管，记忆的难点不是存起来，而是存什么、什么时候写、什么时候读、怎么让它失效；第七步，做评测，按链路拆指标，不能只看最终答没答对。

---

## 核心认知

**Q：AI 应用开发最核心的工程难点是什么？**
**A：** 核心难点不是调模型，而是怎么把大模型的不确定性，通过工程手段变成企业业务可用的确定性。靠数据治理、检索优化、异常容错、权限管控、成本管控这些手段，让 AI 系统稳定、安全、低成本地落地。这恰恰是后端工程师转型 AI 开发最大的优势。
