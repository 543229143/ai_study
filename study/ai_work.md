# AI Agent 面试问答手册

> 基于"非 OSI 新机构接入"项目实战经验整理
> 所有回答均从金融业务场景出发，展示工程落地能力

---

## 一、项目介绍

### Q：介绍一下你做过的 AI Agent 项目？

> "我负责了一个**金融机构新机构接入自动化**项目，把原本需要人工操作的 10 步流程改造成了 AI 驱动的半自动化工作流。包括：预检 → SQL 插入 → 技术文档生成 → GOA 代码 → LCS 代码 → AMS 代码 → LPS 代码 → Mock 文档/上传 → Nacos，共 10 个阶段。"

> "架构上我用了三件套：**SKILL.md**（操作指令集，告诉 AI 每一步做什么）、**Python 脚本**（每个阶段的具体执行逻辑）、**workflow-state.json**（状态文件，记录当前进度，支持中断恢复和错误回滚）。支持全流程（full）、授信导下载（download）、H5 三种工作流类型。"

### Q：你的项目有什么技术亮点？

> "三点：一是**状态驱动的中断恢复**，用 JSON 状态文件记录每个阶段的执行状态，跑崩了可以从指定阶段重跑，不需要从头再来；二是**阶段间解耦**，每个阶段独立可执行，某个阶段失败不影响其他阶段已完成的成果；三是**多工作流支持**，同一套架构支持三种不同的接入流程，通过配置而非改代码切换。"

## 二、Agent 基础认知

### Q：什么是 AI Agent？和普通 API 调用的区别？

> "最核心的区别是**自主决策能力**。普通 API 调用是人告诉系统'做什么、怎么做'；Agent 是你告诉它目标，它自己决定怎么做。比如订机票，普通程序是填表单点提交；Agent 是你说'帮我订去北京的机票'，它自己判断要不要先问你出发城市、调哪个接口、处理异常。"

### Q：Agent 有哪些主流模式？

> "主流的有七种：**

> 1. **ReAct**（思考→行动→观察循环）— 边做边想，适合客服、调试
> 2. **Plan-then-Execute**（先计划后执行）— 路径确定但步骤长，适合数据分析
> 3. **Tool-use Only**（纯工具调用）— 简单单轮，适合查天气+设提醒
> 4. **Multi-agent**（多 agent 协作）— 各司其职，适合复杂业务流
> 5. **Reflexion**（自反思修正）— 执行后自我检查，适合代码生成、写作
> 6. **Tree-of-Thought**（多路径探索）— 同时探索多条路径择优，适合搜索/规划
> 7. **Function Calling**（模型输出 JSON → 程序执行）— OpenAI 默认，最常见

> 它们不互斥，实际项目经常混用。"

### Q：这些模式是谁决定的？Cursor 还是 Codex？

> **模式不是平台决定的，是开发者通过 Prompt 和代码决定的。** 同样是调 LLM：

> - Prompt 写"每次调用前先分析意图" → ReAct
> - 写"先出完整计划再执行" → Plan-then-Execute
> - 写"直接调工具不要思考" → Tool-use Only
> - 写 SKILL.md 按步骤执行 → State Machine

> **Cursor、Codex、OpenAI 都只是执行环境，不是模式的决定者。**"

### Q：怎么判断用哪个模式？

> "一个简单标准：**你知道怎么做**的任务 → 选确定性模式（State Machine/流水线）；**你不知道怎么做、需要现找答案**的任务 → 选动态模式（ReAct/Plan-then-Execute）。前者稳，后者灵活，选错了就会要么僵化要么失控。"

## 三、工具调用

### Q：Agent 是怎么调工具的？需要我告诉它调哪个吗？

> "不需要。你只需要注册工具（给每个工具一个名字和描述），AI 收到用户问题后**自己匹配意图、决定调哪个、自动提取参数**。你不需要写 if-else 判断用户意图，AI 靠 `name` 和 `description` 自己选。"

### Q：工具的参数怎么定义？事前不知道怎么办？

> "参数是你定义**结构**，AI 负责**填值**。你事先写好一个函数（比如 `book_flight(from_city, to_city, date)`），把它的入参原样注册给 AI。用户说'明天从上海到北京'，AI 自动提取参数：`from_city=上海, to_city=北京, date=2026-06-12`。用户没说全的（比如没提出发城市），AI 会自动反问。"

### Q：工具的执行业务逻辑也要我事先写到 Agent 里吗？

> "对，**工具背后的业务逻辑必须你写**。Agent 的自主是有限的——它只能从你注册的工具里选，不能自己发明工具或调用你没注册过的 API。你招了实习生，你给他一个工具箱，他知道用扳手拧螺丝，但扳手是你事先放进去的。"

### Q：MCP 和普通工具调用有什么区别？

> "MCP 改变了工具的**部署位置**，不改变工具调用机制。传统方式工具函数写在 agent 项目代码里；MCP 把工具搬到独立的 Server 上，agent 通过 MCP 协议发现和调用。好处是工具可以**复用和热插拔**——别人写好的 MCP Server 直接连就能用。但业务逻辑还是得你自己写。MCP 不解决'用 ReAct 还是 Plan-then-Execute'的模式问题。"

### Q：写工具 description 有什么技巧？

> "写得越具体，AI 越不容易用错：**

> ```
> ❌ "工具1"       → AI 可能乱用
> ✅ "查询指定城市实时天气，参数 city 城市名，
>     当用户问天气、冷不冷、下不下雨时使用，
>     不用于查询历史天气"
> ```

> 好的 description = **什么时候该用 + 参数含义 + 什么时候不该用**。"

## 四、SKILL.md 方式（你的实战方案）

### Q：你用的 SKILL.md 方式和 Agent 是什么关系？

> "SKILL.md 本质是 **AI 的操作手册**——告诉 AI 每一步做什么、按什么顺序、出错了怎么办。它和 Agent 的区别在于 **控制权在哪**："

> ```
> SKILL.md：人写死每一步 → AI 按菜谱做菜
> Agent：    人给目标和工具箱 → AI 自己决定做什么菜
> ```

> "两者底层能力是共通的（工具调用、状态管理、上下文理解），区别在于**自主决策权的分配**。"

### Q：为什么不用真正的 Agent 模式（比如 ReAct）？

> "**金融合规场景下，可控和稳定是刚需。** 每个步骤都要可追溯、可审计，不允许 AI '突发奇想'跳过或改变步骤。我的架构取舍是：AI 负责**解释执行和异常处理**，流程跳转由状态文件控制。这样既拿到了 AI 的自动化收益（不用人手动执行脚本），又守住了合规底线（流程不可篡改）。"

### Q：那你这算不算 Agent？面试被问"有没有模式"怎么回答？

> "算半自动化 Agent，但不是市面上常见的动态 Agent。如果被问到模式，不要说'没有模式'，要说：**我主动选择了 State Machine 模式**——把流程固化为 SKILL.md，用状态文件驱动阶段流转。这是针对固定流程业务的最优架构选择。"

### Q：SKILL.md 方式有什么局限？

> "三点局限：一是**流程写死后改起来麻烦**，需要改文档；二是**遇到 SKILL.md 没覆盖的异常会卡住**，没有动态应变能力；三是**无法跨任务泛化**，换个业务就得重新写一套 SKILL.md。但这些局限在金融场景下反而是优势——我们恰恰不希望 AI '太灵活'。"

## 五、架构设计

### Q：你觉得一个理想的 Agent 架构是什么样的？

> "我认为是 **混合模式（Hybrid）**——主体流程用确定性状态机保证可控性，关键判断节点插入 LLM 推理做动态决策。比如：预检失败时，AI 自动分析日志、搜索知识库，决定是重试还是跳过还是升人工。这种方案既不会失控，又保留了 AI 的最大价值。"

### Q：如果让你把这个项目升级成真正 Agent，你会怎么做？

> "4 步升级路径：**

> 1. **保持** 确定性阶段框架不变，监管可审
> 2. **替换** 阶段内逻辑：不再写死'先跑 A 再跑 B'，给 AI 一个工具箱让它自主组合
> 3. **引入** 记忆层：状态文件升级为向量数据库，支持跨 session 记忆和知识检索
> 4. **增加** 异常自主处理：失败时 AI 自动诊断，决定重试/跳过/回退/升人工

> 核心思路是 **保持骨架的确定性，替换肌肉的灵活性**。"

### Q：你的 Agent 是怎么调 LLM 的？通过 SDK 吗？

> "对，不管哪种部署方式，调 LLM 都是通过 SDK。Cursor 内嵌的方式是 Cursor 帮我调了 SDK 我不需要写；独立部署的话自己写 SDK 调用。`pip install openai`，`pip install anthropic`，换不同的 `base_url` 就能换不同的模型——OpenAI、Claude、豆包、DeepSeek，SDK 用法基本一样。"

> "SDK 本质就是封装了 HTTP 请求的工具包，帮你把消息和工具注册信息 POST 到 LLM 厂商的 API 上。"

### Q：你的 SKILL.md 方案只能在 Cursor 里用吗？能不能独立部署成 HTTP 服务？

> "可以，这就是我说的 **混合部署模式**。给 SKILL.md 套一层 HTTP 服务就行："

> ```
> 用户 HTTP 请求 → 你的 FastAPI 服务
>   → 代码读 SKILL.md 拼成 system prompt
>   → 你调 LLM SDK（不是 Cursor 调了）
>   → LLM 返回工具调用 → 你执行脚本
>   → 结果回填给 LLM → 生成最终回复
>   → HTTP 返回给用户
> ```

> "核心代码其实很短："

> ```python
> @app.post("/agent")
> def handle(task: str):
>     messages = [
>         {"role": "system", "content": SKILL},
>         {"role": "user", "content": task}
>     ]
>     resp = client.chat.completions.create(
>         model="gpt-4",
>         messages=messages,
>         tools=[...]
>     )
>     return execute_tools(resp)
> ```

> "这样既有 SKILL.md 的确定性流程，又是独立 HTTP 服务，任何人都能调。模型也可以随便换，不再是以前只能在 Cursor 里用的局面了。"

### Q：那这个部署方式里，SKILL.md 起什么作用？

> "它还是当 **system prompt** 用，但阅读者从 Cursor 的 AI 变成了我代码里调用的 LLM。行为逻辑完全复用，只是**谁来读 SKILL.md** 变了：以前是 Cursor 内置的 AI 读，现在是我代码里的 GPT-4/Claude 读。文档不需要改，改的是上层调用层。这也是我当时选择用 SKILL.md 做指令集的价值体现——内容和执行环境解耦。"

### Q：Agent 的安全性怎么保证？

> "三个层面：一是**输入防护**，防 prompt injection；二是**权限控制**，Agent 只能调用已注册的工具，不能越权；三是**关键决策 HITL（Human-in-the-Loop）**，涉及数据删除、资金操作等必须人工确认。在金融场景下，第三点最重要。"
