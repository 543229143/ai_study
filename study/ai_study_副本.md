# AI 学习笔记

## 基础概念

### Tokenizer（分词器）的重要性
- **直接影响 token成本** — 同样内容在不同 tokenizer（分词器）下 token数可能差很多
- **影响跨语言表现** — 不同语言的分词效率不同
- **影响代码表现** — 代码场景下 tokenizer（分词器）的设计对理解至关重要
- **影响上下文利用率** — token数直接影响能处理的上下文长度
- 总结：Tokenizer（分词器）直接决定了成本与效果

### 位置编码（Position Encoding）
- **必须存在的理由**：没有位置信息就无法区分顺序差异
  - 例："我喜欢你" vs "你喜欢我" — 词相同但顺序不同，含义完全相反
- Transformer（变换器）本身是位置不变的（permutation invariant，置换不变性），必须靠位置编码注入位置信号
- 常见方案：RoPE（旋转位置编码）、ALiBi（线性偏置位置编码）等

### 幻觉（Hallucination）根因分析

**底层根因**：
1. **目标函数不是事实核验器** — 模型训练目标是预测下一个 token，不是验证事实真伪
2. **训练数据本身就有噪声、过时与冲突**
3. **上下文证据不足或上下文组织不当**：
   - 检索召回错了文档
   - 关键信息埋在长上下文中间（lost in the middle）
   - 多个版本的业务规则同时出现
   - 系统提示要求必须回答，禁止说"不确定"
   - 这种情况下，模型"编"出来的内容，某种意义上**是系统设计在逼它编**

**采样与累积误差**：
- 采样参数越发散（temperature（温度）高），模型选择低概率 token的机会越高
- 多步生成越长，局部小错越可能滚成整体大错
- 特别是 **长文本、复杂 SQL、长代码、工具串联计划** — 一步偏了，后面都跟着漂移

**Agent（智能体）时代的幻觉新形态**：
- 模型错误理解了工具返回
- 工具本身返回脏数据
- 模型声称工具执行成功，实际上并没有
- 模型把未发生的外部动作说成已完成
- Anthropic Sonnet 4.6 特别提到 "fewer false claims of success"（更少虚报成功） — 在 Agent（智能体）/ 代码场景里，**"虚报完成"已经成为一种非常实际的幻觉类型**

### 降低幻觉：方法分层

**第一层：任务定义层（最常见也最被忽略）**
- 如果任务允许"不知道"，就**明确允许**
- 如果需要证据，就要求**"先找证据再回答"**
- 如果必须引用来源，就把 citation（引用）/ evidence slot（证据槽位）**放进输出 schema（模式）**
- 如果问题时间敏感，就要求**调用搜索或检索**
- 核心：让模型知道 **没有证据时承认不确定，是被允许甚至被鼓励的**

**第二层：上下文与检索层**
- 降低幻觉最有效的方法之一，往往不是"改模型"，而是**给更好的证据**
  - 提升检索召回与重排质量
  - 减少上下文噪声
  - 把关键证据放在更显著位置
  - 用结构化字段而不是大段原文堆叠
  - 对版本号、日期、实体名做显式突出
- Anthropic reduce hallucinations 文档也强调 **"evidence-first"（证据优先）策略** — 先提取证据、再作答，在长文档问答中很有用
- Agent（智能体）场景更要控制 side effects（副作用）

**第三层：输出约束层**
- 结构化输出不会自动提高真实性，但可以**强迫模型把"答案"和"依据"拆开**
- 有效的 schema（模式）设计：
  ```
  answer
  confidence / uncertainty note（置信度 / 不确定性说明）
  evidence_ids（证据 ID）
  cannot_answer_reason（无法回答的原因）
  ```
- 当模型必须显式写出依据来源时，至少更容易被程序和评测发现它在编

**第四层：外部验证层**
- 对时间敏感、事实敏感、高风险任务，不能只靠参数知识
  - 调搜索、查数据库、走代码执行
  - 做二次校验
  - 必要时让 verifier（验证器）或 rule engine（规则引擎）复核

**第五层：数据闭环与持续监控**
- 真正生产中幻觉治理离不开数据闭环：
  - 哪些问题类型最容易编
  - 哪些来源最不可靠
  - 哪些 prompt（提示词）促使模型过度自信
  - 哪些工具经常被误用
  - 哪些版本升级后出现回归
- **幻觉不是一次性解决，而是持续监控与回归测试对象**

> **幻觉不是模型"坏习惯"，而是 next-token 目标、数据噪声、证据不足和系统设计共同作用下的自然结果。**
>
> **Alignment 不只是让模型更安全，更是让模型更像一个可控、诚实、能遵守优先级的系统组件。**
>
> 降低幻觉不是靠一个更聪明的模型，而是靠**证据、验证和工作流设计**。

### 常见误区

1. **把所有错误都叫幻觉** — 检索错、工具错、路由错、格式错、指令冲突需要分开治理
2. **以为更强指令跟随天然会减少幻觉** — 错误目标下的强执行，反而可能放大幻觉
3. **把 alignment（对齐）等同于"安全拒绝"** — 对齐同样包括 helpfulness（有用性）、格式稳定、边界控制与诚实表达不确定性
4. **只靠 prompt（提示词）解决所有幻觉** — 真正高风险任务必须引入外部证据和验证闭环

### 幻觉面试问答

**Q1：什么是幻觉？**
模型生成了缺乏证据支持、与事实或给定上下文不一致、但表面流畅合理的内容。

**Q2：幻觉的根因是什么？**
最底层是训练目标不是事实核验，而是 next-token prediction（下一个词元预测）；再叠加数据噪声、证据不足、任务约束不清、解码随机性和系统外部错误。

**Q3：对齐（Alignment）为什么重要？**
因为大模型不只是要"会"，还要"好用、可控、守边界、按格式、在不确定时诚实"。对齐提升的是可用性与可控性。

**Q4：指令跟随越强越好吗？**
不绝对。强指令跟随需要正确的优先级和边界设计，否则可能更积极地执行错误或有害指令。

**Q5：为什么说"不要乱编"本身也是一种 prompt（提示词）设计？**
因为你需要明确允许模型承认不知道，并把缺证据时的行为定义为合法输出，而不是逼它强答。

**Q6：如何降低幻觉？**
从任务定义、证据检索、上下文组织、结构化输出、工具验证、评测监控六个层面共同治理，而不是只换模型。

**Q7：模型引用了来源就一定没幻觉吗？**
不一定。它可能编造来源、错引段落、误解证据。引用只是辅助约束，不是事实保证。

**Q8：Agent 场景里的幻觉有什么新表现？**
包括虚报任务完成、误报工具执行成功、误解工具返回、把计划当结果、把模拟输出当真实外部状态。

**Q9：什么时候应该让模型拒绝回答？**
当问题高风险、缺乏可靠证据、请求越权或有明显安全边界冲突时。拒绝本身也要设计成 informative refusal（有信息量的拒绝），而不是机械回避。

### LLM（大语言模型）中的知识存储
- **以分布式参数形式压缩在模型权重中**，而非可查询的数据库
- 因此会存在以下问题：
  - **知识会过时** — 训练数据截止后无法自动更新
  - **容易混淆** — 参数式存储天然存在模糊性
  - **细粒度事实易出错** — 低频或细节性事实不可靠
- **为什么需要 RAG（检索增强生成）/ 工具调用 / 检索验证**：
  - RAG（Retrieval-Augmented Generation，检索增强生成）引入外部知识库弥补参数化知识的不足
  - 工具调用让模型实时获取最新数据
  - 检索验证帮助检测和纠正事实错误

## 推理机制

### 推理的三种形态
| 概念 | 说明 |
|------|------|
| **显式推理文本**（Chain-of-Thought，思维链） | 模型把思考过程写出来给你看 |
| **内部推理预算**（Reasoning Budget） | 模型在内部花更多计算，但不一定完整暴露过程 |
| **最终可见答案** | 真正返回给用户或程序消费的结果 |

- **关键认知**：很多时候产品并不需要完整思维链暴露，只需要更高正确率、更稳结构和更低误答率
- **面试经典问题**："要不要让模型把思考过程全吐出来？"
  - 更稳的回答：**不一定**
  - 需要结合 **安全、稳定性、用户体验、成本** 综合决定
  - 很多生产场景更关心 **reasoning budget（推理预算）与最终输出质量**，而不是完整中间文本

### 指令层级（Instruction Hierarchy）

**指令不是一层，而是多层。** 真实系统里的指令来源至少包括：
1. **平台/模型层原则** — 模型内置的安全和行为约束（最高层）
2. **system/developer 指令** — 开发者定义的系统提示
3. **用户指令** — 用户的直接输入
4. **外部工具或检索内容** — 工具返回、检索到的文档
5. **历史对话中的旧约束** — 前几轮对话中给出的要求（最低层）

**关键问题**：这些层的指令可能冲突。如果模型没有优先级意识，就会被低优先级文本带偏。

### 推理模型的最佳实践（大模型 vs 小模型）

**大模型（强推理模型）**：
- 设计上已经会进行内部分析，不需要过度指导
- **不要**：给过于繁复、过细、甚至相互冲突的步骤指令 → 反而可能限制模型
- **应该**：把**目标、约束、验收标准、可用工具、输出格式**讲清楚
- 把"怎么想"留给模型自己完成

**小模型（mini/nano 级别或不擅长推理的模型）**：
- **需要更强的约束** — 更明确地拆任务
- **更直接的格式要求** — 避免歧义
- **提供 few-shot（少样本）示例** — 更接近真实分布的样例
- **避免隐含要求** — 不要一个 prompt（提示词）里塞太多隐含要求
- **更短上下文** — 减少无关信息干扰
- **更直接 schema（模式）** — 结构化输出定义要清晰

**面试经典问题**：同一个 prompt（提示词）在大模型和小模型上要怎么改？
- **标准答案**：小模型需要**更强约束、更少歧义、更短上下文、更直接 schema（模式）**

**业界趋势**：越来越强调 **明确目标和边界**，而不是把中间思路写得极其死板

### Prompt Engineering（提示工程）

**定义**：围绕模型输入进行**设计、约束、分层、示例化和迭代**的过程，其目标是在**给定模型与上下文预算**下，**稳定地**实现特定任务结果。

三个重点：
1. **"围绕模型输入"** — prompt 不只是用户输入，还可能包含 system 指令、developer 规则、few-shot（少样本）示例、输出 schema（模式）、工具说明、检索证据等
2. **"稳定实现"** — 目标不是偶尔跑出一个好答案，而是在真实流量里**稳定、可复现、可评估**地完成任务
3. **"给定模型与上下文预算"** — prompt 不能脱离模型家族、token 成本、上下文长度、结构化输出能力、工具协议单独讨论。一个在大模型上可行的 prompt，未必适合 mini/nano 模型；一个在短上下文下有效的 prompt，未必适合 Agent（智能体）多轮链路

### 强 prompt 的六类信息

1. **任务目标** — 模型到底要做什么。目标必须可操作，不能只是"帮我处理一下"。差的："请分析下面内容。" 好的："请从下面病历文本中抽取患者姓名、主诉、诊断、用药，并输出为 JSON。"
2. **输入边界** — 区分哪部分是输入、哪部分是规则、哪部分是示例、哪部分是背景。分隔符、标题、XML/JSON 包裹、Markdown heading 等都属于输入边界设计。很多模型失败不是不会做，而是没分清"正文"和"指令"
3. **约束条件** — 不要输出什么、不允许编造什么、信息不足怎么办、是否允许调用工具、是否严格按 schema 输出、是否先提取证据再回答。约束越明确，模型越不容易自由发挥到错误方向
4. **输出格式** — 自然语言 / Markdown / 表格 / JSON / 函数参数 / 多段结构。这是最常被忽略却最容易直接提升工程可用性的部分，没有格式约束则后续解析和自动化都很难稳定
5. **示例与反例** — few-shot 不是"给模型看几个漂亮答案"，而是告诉模型任务分布、边界处理、输出风格、哪些答案算错。尤其在小模型、抽取、分类、结构化输出任务里，示例常比空泛指令更有用
6. **自检与停止条件** — 输出前检查：是否覆盖所有必填字段、是否所有结论有证据、是否工具结果已回填、是否输出合法 JSON。这类轻量自检比单纯写"请仔细思考"更有效

### Prompt Engineering 核心原则

**原则 1：把目标写成可验收的任务**
- 目标对象、成功标准、不允许事项、输出格式
- "写得更专业一点"这种描述不够可操作

**原则 2：减少隐含假设**
- 模型擅长补全，但不代表隐含假设都会被正确补全
- 读者、语气、长度、字段、格式、失败边界不说清楚，模型只能猜
- Prompt 工程本质之一，就是把脑子里的假设尽量显式写出来

**原则 3：分离规则、数据和示例**
- 分块写：规则区 / 输入数据区 / 示例区 / 输出区
- 效果更稳，也更容易维护和测试

**原则 4：把"不知道怎么办"也写进去**
- 若证据不足 → 返回 cannot_answer
- 若字段缺失 → 返回 null 而不是编造
- 若工具失败 → 输出 error_reason
- 若请求越权 → 拒绝并说明原因
- 好的 prompt 不只定义成功路径，也定义失败路径

**原则 5：Prompt 是要测的，不是要信的**
- 用代表性样本测：任务成功率、格式合法率、幻觉率、工具调用准确率、平均 token/延迟、边界输入鲁棒性
- Prompt engineering 本质上是 **eval-driven iteration（评估驱动的迭代）**

### 常用 Prompt 模板思路

**1. 抽取型模板** — 信息抽取、简历解析、票据解析、工单解析
- 关键要素：字段列表、字段含义、缺失时输出 null、不允许编造、输出 JSON Schema
- 通常温度更低、格式约束更强

**2. 证据问答模板** — RAG、文档问答、知识库问答
- 关键要素：只能依据给定证据回答、先定位证据再形成结论、若证据不足明确返回无法回答、最终附 evidence_ids / source list
- 对减少幻觉特别有效

**3. 工具调用模板** — Agent、函数调用、自动化工作流
- 关键要素：工具说明、调用时机、输入参数约束、工具失败处理、禁止在未调用工具时假装已执行
- 对降低"虚报已完成"非常关键

**4. 评审/打分模板** — LLM-as-a-Judge、简历评分、内容审核
- 关键要素：评分维度、评分等级定义、先列观察再给分、输出 JSON 或表格、禁止越过给定标准自由发挥
- 评分任务最怕标准漂移，prompt 里必须把 rubric 写清楚

### Prompt Engineering 高频面试问答

**Q1：Prompt Engineering 到底是什么？**
不是"和模型聊天的技巧"，而是围绕模型输入进行任务定义、约束、示例化和迭代的工程过程。

**Q2：好的 prompt 最重要的部分是什么？**
最重要的是明确任务目标、约束边界和输出格式，而不是把 prompt 写得太长或太像自然对话。

**Q3：Few-shot 一定比 zero-shot 好吗？**
不一定。强模型在常规任务上 zero-shot 已经很好；few-shot 更适合边界复杂、标签容易混淆、输出格式特殊、小模型场景。

**Q4：为什么 prompt 要分块写？**
因为规则、数据、示例、输出区分清楚后，模型更容易正确解释输入，也更容易维护和调试。

**Q5：如何降低 prompt 导致的幻觉？**
显式写明证据边界、缺证据处理方式、字段缺失输出规则，并尽量提供结构化证据和输出 schema。

**Q6：为什么不能只靠 prompt 解决一切？**
因为模型能力、上下文质量、工具设计、缓存预算、评测闭环同样重要。prompt 是大杠杆，但不是唯一杠杆。

**Q7：reasoning model 的 prompt 和普通模型有什么不同？**
复杂 reasoning model 更适合明确目标和验收标准，而不是过细 micromanage；小模型则更需要显式步骤和例子。

**Q8：prompt 改动为什么要配合评测？**
因为它本质上会改变模型行为，需要用数据看成功率、鲁棒性、格式合法率和成本是否真的变好。

**Q9：Prompt Engineering 和 Context Engineering 的区别是什么？**
前者更偏单次提示设计与任务表达，后者更偏整个运行时上下文的选择、压缩、排序和生命周期管理。

**Q10：项目里 prompt 一般怎么迭代？**
先定义数据集与指标，分析失败类型，再定向修改任务描述、约束、示例或结构化输出要求，而不是盲改措辞。

### Context Engineering（上下文工程）

**定义**：围绕模型运行时输入进行**选择、分层、排序、压缩、生命周期管理和验证**的工程过程。

六个关键词：
1. **选择** — 哪些内容应该进当前轮上下文
2. **分层** — 系统规则、任务目标、证据、示例、工具、状态要不要分区
3. **排序** — 什么靠前，什么靠后
4. **压缩** — 哪些历史要总结，哪些原文要裁剪
5. **生命周期管理** — 哪些信息只在当前轮有效，哪些应跨轮保留
6. **验证** — 输入给模型的上下文是否完整、冲突、过时或越权

> Prompt Engineering 更像在优化**"怎么说"**，而 Context Engineering 更像在优化**"让模型在这一刻看到什么、以什么结构看到"**。

### Prompt Engineering vs Context Engineering

| 维度 | Prompt Engineering | Context Engineering |
|------|-------------------|-------------------|
| 定位 | 偏**表达与约束** | 偏**运行时状态组织** |
| 关注点 | 任务怎么描述、规则怎么写、输出格式怎么限定、few-shot 示例怎么给、工具说明怎么表述 | 本轮需要哪些外部证据、历史保留到什么程度、用户记忆是否注入、工具结果原样放还是摘要放、哪些常驻哪些按需加载、多轮长任务怎么做 compaction |
| 本质 | **单轮输入文本与 schema 的设计** | **整个运行时输入系统的设计** |

**两者不是替代关系**：
- 没有好的 prompt，context 再好也会执行歪
- 没有好的 context，prompt 再漂亮也会在错误材料上发挥
- **真正生产里，两者必须一起设计**

### Context 的组成

| 类型 | 说明 |
|------|------|
| **规则上下文** | system / developer prompt、产品规则、权限边界、品牌风格、合规要求、审批策略。通常优先级最高，适合做稳定前缀 |
| **任务上下文** | 用户当前请求、本轮输入数据、UI 状态、用户选中的文件、当前页面信息、表单字段。最直接的任务驱动信息 |
| **知识上下文** | 检索召回的文档、FAQ、业务规则、数据库结果、网页搜索结果、代码片段。决定模型有没有事实依据 |
| **状态上下文** | 会话历史、计划进度、已完成步骤、待办事项、失败重试次数、上次工具返回、临时变量。Agent 系统特别依赖这类 |
| **记忆上下文** | 长期用户偏好、组织信息、项目约定、个人习惯、长期目标。不是每轮都变，但也不该每轮全部注入 |
| **工具上下文** | 工具名字、用途、参数 schema、权限边界、调用代价、返回格式、错误定义。工具越多，越容易成为上下文膨胀源 |

**解决方案三件事**：
1. **分区标注** — 把规则、证据、状态分开标注，模型能区分各自作用
2. **结构化压缩** — 长会话历史压成 structured summary / state（结构化摘要/状态），而不是全量回灌
3. **Just-in-time 加载** — 对大文件和大工具集改成按需注入，只在需要时才加载

**效果**：复杂任务稳定性明显提升，token 成本和延迟也降了。

### Context Engineering vs RAG vs Memory

| 概念 | 职责 |
|------|------|
| **RAG** | 负责**"找"** — 提供知识来源 |
| **Memory** | 负责**"记"** — 保留跨轮持续信息（用户偏好、长期目标等） |
| **Context Engineering** | 负责**"在这一轮怎么用"** — 决定知识是否该用、如何拼接、如何压缩、放在哪里 |

**RAG 不是全部 context engineering** — RAG 偏"如何拿到知识"，而 context engineering 还包括：是否需要这个知识、用原文还是摘要、放什么位置、和规则/历史/工具结果怎么拼、是否只保留引用而不展开。

**Memory 也不是全部 context engineering** — Memory 关注跨轮信息，而 context engineering 决定：哪些记忆该在本轮注入、以什么粒度注入、何时不要注入、记忆与实时输入冲突时谁优先。

### Context Engineering 高频面试问答

**Q1：什么是 context engineering？**
是对模型运行时输入进行选择、排序、压缩、生命周期管理和验证的工程过程。

**Q2：它和 prompt engineering 的区别是什么？**
Prompt 更偏单轮任务描述与约束表达；context engineering 更偏整个运行时上下文系统如何组装和管理。

**Q3：为什么现在越来越强调 context engineering？**
因为模型和 Agent 的输入来源变多，失败往往来自错上下文、旧状态、噪声和冲突，而不仅是提示词措辞。

**Q4：上下文越多越好吗？**
不一定。过多上下文会增加噪声、成本和延迟，还可能降低关键信息密度。

**Q5：如何提高上下文质量？**
确保相关性、权威性、新鲜度，减少冲突和冗余，关键内容显式标注并放在更显著位置。

**Q6：RAG 和 context engineering 有什么关系？**
RAG 提供知识来源，context engineering 决定这些知识是否该用、如何拼接、如何压缩、放在哪里。

**Q7：多轮 Agent 为什么需要状态压缩？**
因为原始历史会快速膨胀并污染上下文，应该把历史转成结构化状态或 summary。

**Q8：什么是 just-in-time context？**
先保留轻量引用，在模型真正需要时再加载详细内容，避免预先塞入大量可能无关的信息。

**Q9：为什么要分区标注上下文？**
为了让模型区分规则、证据、示例、记忆和工具结果，降低误读输入结构的风险。

**Q10：怎样判断一个问题是 prompt bug 还是 context bug？**
如果目标和约束表达已经清晰，但模型在错误材料上推理、混用旧状态或无视证据，通常更像 context bug。

**Context Engineering 常见误区**：
1. 以为 context engineering 只是"多加一点背景资料" — 其实它更强调**取舍、排序和生命周期**
2. 把历史消息全量保留当成"信息更完整" — 现实里这常常意味着**更脏、更慢、更冲突**
3. **不区分规则与证据** — 模型会把它们混读，导致既不听规则，也不忠于证据
4. 认为有了大上下文就不用做 context engineering — 恰恰相反，**窗口越大，选择和压缩越重要**

> Context engineering 不是"给更多信息"，而是"在这一轮给最对的信息，以最合适的结构给出去"。

### 采样参数 vs 推理预算
- **采样参数**（temperature（温度）/ top_p（核采样阈值）/ max_tokens（最大词元数））：控制**怎么从概率分布里拿答案**
- **推理预算**（Reasoning Budget）：控制模型在拿答案前**愿意花多少计算去想**
- 两者都重要，但**都不能替代正确的任务建模**
- **生产上更有效的优化顺序**：
  1. **模型选型** — 选对模型比调参数更重要
  2. **上下文证据** — 提供充分、准确的参考信息
  3. **输出约束** — 定义清晰的格式和 schema（模式）
  4. 最后再调 **temperature（温度）/ top_p（核采样阈值）/ max_tokens（最大词元数）** 等采样参数

### Temperature（温度）
- **定义**：控制输出概率分布的平滑程度，决定模型的随机性 / 确定性
  - **Temperature = 0** → 总是选最高概率 token（确定性，重复性高）
  - **Temperature 低**（如 0.1~0.3）→ 偏向高概率 token，输出稳定，适合事实性 / 代码任务
  - **Temperature 高**（如 0.8~1.5）→ 概率分布更均匀，输出更多样 / 有创意，适合写作、头脑风暴
  - **Temperature 极高**（> 1.5）→ 几乎随机输出，容易乱说
- **不能"就靠调 temperature（温度）解决问题"**：
  - Temperature（温度）只是采样策略的一个旋钮，不是万能药
  - 真正的问题需要从模型选型、Prompt（提示词）设计、RAG（检索增强生成）、微调、工具调用等根本上解决
  - 调 temperature（温度）更多是输出风格微调，无法弥补模型能力或知识上的不足
- **Temperature = 0 模型就不会错了吗？→ 不是**
  - Temperature=0 只是让采样完全确定（每次都选最高概率 token），但依然会犯错
  - 模型本身的**知识不完整或有偏差** — 训练数据里的幻觉和偏见
  - **最高概率路径也可能是错的** — 模型可能对错误答案非常有信心
  - 输出质量还受 **prompt（提示词）设计、上下文长度、模型能力天花板** 限制
  - Temperature=0 能消除随机性，但消除不了模型的固有缺陷

## 结构化输出与 Function Calling

如果说 Prompt Engineering 是让模型更**"会做题"**，那么结构化输出与 function calling（函数调用）则是让模型真正**"接进系统"**。

### Function Calling 本质流程（五步）

```
用户请求 → 模型 → 工具调用建议 → 应用代码执行工具 → 工具结果回传 → 最终回答
```

1. **把可用工具与 schema 发给模型**
2. **模型返回工具调用建议**（选哪个工具 + 带什么参数）
3. **应用侧代码执行工具**（真正执行业务动作，模型不直接调外部 API）
4. **把工具结果回传给模型**
5. **模型给出最终答案，或继续提出新的工具调用**

> 这五步里，真正容易出问题的不是第 2 步，而是**第 3 步和第 4 步之间的衔接**。执行环境、权限系统、重试策略、幂等设计没做好，模型再聪明也救不了系统。

**三种模式对比**：

| 模式 | 流程 |
|------|------|
| **普通文本输出** | 模型 → 人读 |
| **结构化输出（JSON）** | 模型 → JSON Schema |
| **Function Calling** | 模型 → 工具调用建议 → 应用代码执行 → 工具结果 → 模型继续 |

### 20 行核心 Agent 循环

Step 1: 用户问"哪张专辑曲目最多" → 发给 LLM（带 tools 定义）
Step 2: LLM 返回 tool_calls → 说调 ask_database，参数是 SQL 查询
Step 3: 你执行 SQL → 把结果塞回 messages（role: "tool"）
Step 4: 再调一次 LLM → LLM 用数据库结果生成自然语言回答

核心循环就是：

```python
while True:
    response = client.chat.completions.create(model=model, messages=messages, tools=tools)
    msg = response.choices[0].message
    if msg.tool_calls:                # LLM 想调函数
        for tool in msg.tool_calls:
            result = execute_function(tool.function.name, tool.function.arguments)
            messages.append({"role": "tool", "content": result})
    else:                              # LLM 给出最终回答
        print(msg.content)
        break
```

LLM **不负责执行函数**，它只负责**决定调哪个函数 + 填参数**。真正执行的是你的代码。

### 面试官考察要点

1. **输出合法 JSON ≠ 输出符合 schema 的 JSON**
   - JSON mode 只保证是有效 JSON，不保证符合给定 schema
   - Structured Outputs 才是让模型严格对齐 schema（字段名、类型、枚举值、必填项都稳定一致）
   - OpenAI（2024）、Anthropic（2026）、Gemini 均已 GA

2. **Function Calling 本质是模型规划 + 程序执行**
   - 模型输出一个建议调用哪个工具、带什么参数的结构
   - 真正执行业务动作的是应用代码，不是模型自己直接跑外部世界

3. **生产思维**
   - schema 要严格、参数要验证
   - 高风险动作要审批、工具失败要回传
   - 外部副作用要幂等
   - **不允许模型在没执行时谎称执行成功**

### Function Calling 关键工程点

```
┌─────────────────────────────────────────────────────────────┐
│                    模型（LLM）                              │
│  输出: tool_name + params (如 get_order(id=123))            │
└────────────────────┬────────────────────────────────────────┘
                     │ function calling 协议层
                     ▼
┌─────────────────────────────────────────────────────────────┐
│                  Orchestrator（编排层）                      │
│  职责: 接收模型建议 → 路由 → 校验 → 执行 → 回传结果          │
└────┬─────────┬──────────┬──────────┬───────────────────────┘
     │         │          │          │
     ▼         ▼          ▼          ▼
┌────────┐ ┌────────┐ ┌────────┐ ┌────────────┐
│ 本地   │ │ HTTP   │ │ MCP    │ │ 消息队列    │
│ 函数   │ │ API    │ │ Server │ │ / Shell等   │
│ 调用   │ │ REST   │ │        │ │            │
└────────┘ └────────┘ └────────┘ └────────────┘
     │         │          │          │
     ▼         ▼          ▼          ▼
┌─────────────────────────────────────────────────────────────┐
│                 执行结果（结构化返回）                       │
│  error_code / retryable / data / human_message              │
└────────────────────┬────────────────────────────────────────┘
                     │ 回传给模型
                     ▼
┌─────────────────────────────────────────────────────────────┐
│                    模型（LLM）                              │
│  根据结果决定: 继续调工具 / 给出最终答案                      │
└─────────────────────────────────────────────────────────────┘
```

**核心分层**：
- **模型层** — 只负责"建议"（tool_name + params），不负责执行
- **协议层** — Function Calling 接口协议（模型怎么表达调用意图）
- **编排层** — 接收建议、校验、路由到正确执行方式、回传结果
- **执行层** — 本地函数 / HTTP / MCP 等各种执行方式，模型不知道具体走哪个
- **回传层** — 结果结构化回传，让模型决定下一步

**1. 模型只决定建议调用，程序决定是否执行**
- 模型建议 `transfer_money(amount=10000)`，但真正是否转账由程序做**权限校验、额度校验、审批校验**
- 面试中能明确这句，基本避开了 function calling 最大的认知坑

**2. 工具描述要清晰，但不要冗长**
- 坏：工具名模糊（如 `process_data`）、描述太短或太长
- 好：清楚说明用途、输入、边界和限制，不是堆 SDK 文档

**3. 工具要做能力最小化**（安全相关）
- 高风险动作拆成多步：`get_balance` → `create_transfer_draft` → `confirm_transfer`
- 比单一 `transfer_money` 更安全、更好控

**4. 高风险动作必须有 approval / human-in-the-loop（人工介入）**
- function calling 的可靠性不只靠 schema，还靠**审批流和 side-effect guardrail（副作用护栏）**

**5. 工具失败必须结构化回传**
- 不能只返回自然语言"出错了"
- 应返回：`error_code`（错误码）/ `retryable`（可否重试）/ `human_message`（人工消息）
- 让模型决定重试、降级、换工具还是请人工介入

**6. 生产级防错**
- **校验先于执行** — schema 校验 → 类型校验 → 权限校验 → 业务规则校验，通过后再执行
- **幂等与重试** — 创建订单、发消息、扣费必须幂等（idempotent）
- **显式记录 tool trace（工具追踪）** — 记录模型为什么选这个工具、参数、返回结果、最终答案
- **不要把自由文本塞进高风险参数** — 能用 enum、ID、日期的地方就不要让模型自由生成
- **最终答案要对齐工具真实结果** — 最终回答必须与工具真实返回绑定，不能凭模型主观推断

> **结构化输出**解决的是模型结果怎么稳定接进程序，**Function Calling** 解决的是模型如何以受控方式请求外部能力。
>
> 模型负责提出动作意图，程序负责校验与执行；**高风险副作用绝不能直接交给模型自由决定。**

### 工具 Schema 注入方式

| 方式 | 说明 | 适用场景 |
|------|------|----------|
| **全量注入** | 所有工具 schema 一次性放请求里 | 工具少（<20 个） |
| **按需动态注入** | 代码先判断意图，只注入相关工具 schema | 中等数量工具，可区分 domain |
| **MCP（Model Context Protocol）** | MCP client 统一管理工具注册、发现、调用，模型只需一个接口 | 标准化多工具管理 |
| **Tool Search / 工具搜索** | 模型先在索引搜到相关工具，再加载 schema 调用（Anthropic 方案） | 数百上千个工具 |
| **分层路由** | 先给一级路由工具，模型输出意图后，再注入二级具体工具 schema | 超大规模工具集 |

> 实际生产往往混合使用：核心工具全量注入 + 低频工具按需注入 + 超大工具集走 tool search。

### 工具 Schema 怎么设计（高频追问）

**1. 工具名称要明确**
不要叫 `do_task`、`process_data` 这种含糊名字。工具名应该让模型一眼知道边界，例如 `search_public_web`、`query_customer_order`、`create_refund_request`、`send_email_draft`。

**2. 描述要强调边界和适用条件**
工具描述不应只是"用于搜索"。更好的写法是说明：什么时候该用、什么时候不该用、参数来源是什么、是否有副作用、是否需要审批。

**3. 参数要尽量结构化**
不要把一堆自由文本塞进一个 input 字段。越结构化，模型越容易稳定生成，程序也越容易校验。例如把邮件发送拆成 `to`、`cc`、`subject`、`body`、`require_approval`，而不是一个大字符串。

**4. 把约束前置到 schema**
- 如果某个字段只能取枚举值，就不要让模型自由发挥
- 如果某字段必须是 ISO 日期格式，就明确约束
- 如果某动作只允许当前租户资源，就在参数设计或执行层强约束，而不是靠提示词提醒

**5. 明确副作用和幂等要求**
例如：发送类动作是否支持 dry-run（试运行）；创建类动作是否要求 client_request_id；更新类动作是否支持 version check；删除类动作是否必须人工确认。

> schema 不是给模型看的说明书，它同时也是给执行器做安全控制和参数校验的合同。

**Structured Outputs 和 Function Calling 怎么选？**
如果我要的是最终答案的结构，用 Structured Outputs；如果我要的是让模型触发外部能力，用 Function Calling。前者约束输出，后者组织执行。

**A2A（Agent-to-Agent，智能体间通信）和 MCP 的区别是什么？**
MCP 解决的是 agent-to-tool / app-to-context 的标准接入问题，A2A（Agent-to-Agent）解决的是 agent-to-agent 的标准协作问题。前者接能力，后者协同工作。

### Action Execution：为什么执行器不是简单调个 API

工具设计得再好，没有执行器治理也不行。执行器本质上是 agent 和现实世界之间的安全门。

**1. 幂等性**
对有副作用的动作，比如发邮件、创建工单、下单、记账，必须考虑重复执行。常见做法包括：client_request_id、幂等键、事务日志、写前去重、重试时只允许在幂等窗口内重放。

**2. 重试策略**
不是所有错误都适合自动重试。需要区分：
- **瞬时错误**：如网络超时、限流，可重试
- **参数错误**：不应盲目重试，应修正参数
- **权限错误**：需要升级权限或人工介入
- **业务规则错误**：应停止并反馈

**3. 超时与取消**
长动作如果没有超时和取消机制，agent 很容易被卡死。特别是外部 API、浏览器、代码执行、文件读写，都需要时间边界。

**4. 事务与回滚**
对多步副作用动作，最好能设计事务边界。比如创建草稿→上传附件→发送，如果中间某步失败，是否可回滚到草稿态？如果完全不可回滚，就要在更早阶段做审批和 dry-run。

**5. 沙箱与资源隔离**
对代码执行、文件系统、浏览器操作，更要限制：网络访问、文件范围、CPU/内存、可调用命令、持久化能力。

### 并行 Tool Use：收益很大，但不是免费午餐

**并行收益**：降低总时延、更适合独立数据源抓取、提高整体吞吐、适合 DAG 型任务执行。

**并行风险**：参数依赖关系搞错会导致错误结果、副作用动作并行可能引发一致性问题、上下文中很难同时消费过多并发结果、并发过高会引发限流、成本飙升和调试困难。

**适合并行**：多来源检索、多文档摘要、独立 API 查询、批量数据分析。

**不适合并行**：有强顺序依赖的任务、会修改共享状态的动作、需要人工审批的关键路径、高风险事务链路。

### 面试高频点：如何防止工具误调用和越权

**1. 最小权限原则** — 每次任务只暴露必要工具，不要把整个工具宇宙都交给模型。

**2. 工具分级** — 分为只读低风险、写操作中风险、高风险破坏性操作，不同级别对应不同审批和日志要求。

**3. 参数白名单与租户隔离** — 不仅要校验字段类型，还要校验是否属于用户有权访问的资源范围。

**4. 审批前置** — 对于发送、支付、删除、发布等动作，提供 dry-run 和人工确认。

**5. 二次确认与结果复核** — 某些操作即使执行成功，也要要求 verifier 再判断结果是否合理。

**6. 审计记录** — 能回答：谁触发任务、模型选了哪个工具、参数是什么、程序校验后变成什么、最终执行结果、是否有人类审批、是否发生重试或回滚。

### Tool Use 与 Prompt / Context Engineering 的关系

这里非常容易答浅。成熟回答应该是：

1. **Prompt** 决定模型是否知道什么时候用工具
2. **Context** 决定模型能不能拿到足够信息正确填参数
3. **Schema** 决定模型输出是否稳定可校验
4. **Executor** 决定动作能否可靠、安全地落地
5. **Verifier（验证器）** 决定系统是否知道这一步到底做成了没有

> Tool Use 绝不是加一个 function calling 开关，而是从 prompt 到 runtime 的全链路设计。

### 典型失败模式

**失败模式一：工具太多，模型不会选**
解决：按任务裁剪工具集，做分层路由。

**失败模式二：工具描述含糊，模型乱填参数**
解决：重写 schema、增加示例、减少自由文本字段。

**失败模式三：程序对参数完全信任**
解决：执行前强校验，不可信输入全部经过正规化。

## Memory（记忆系统）

从系统角度看，memory 解决的是：系统如何在跨步骤、跨回合、跨任务甚至跨天的执行中，保留对后续决策真正有帮助的信息。

如果没有 memory，agent 每次都像刚被创建出来的新个体，只能依赖当前上下文；
如果 memory 设计得不好，agent 又会变得迟钝、污染、过时，甚至在错误经验上越走越偏。

因此，memory 的难点从来不是存起来，而是**存什么、何时写、何时读、如何失效、怎样避免污染**。

### Memory vs Context vs State vs RAG

这是面试里必讲的边界。

**1. Context**
Context 是当前模型调用时实际送进 prompt 的那部分信息。它是当下可见窗口，可能来自系统提示词、用户输入、工具结果、摘要、memory 检索结果等。所以 context 是模型眼前看到的东西，不等于 memory 本身。

**2. State**
State 更偏运行时状态。它包括当前任务 ID、阶段进度、已完成步骤、预算消耗、临时变量、中间产物、审批状态等。State 不一定长期保存，也不一定适合被称为 memory。

**3. Memory**
Memory 更强调对未来决策有价值的可延续信息。它可以来自历史对话、历史任务、用户偏好、成功经验、失败经验、总结笔记、长期知识片段等。换句话说，memory 是可被未来再次利用的过去。

**4. RAG**
RAG 主要解决外部知识召回问题。它关心的是：面对当前问题，从知识库或文档库里取回最相关的信息。RAG 的对象通常是相对客观、共享的外部知识；memory 的对象更偏个体化、任务化、经验化的信息。

> RAG 面向外部知识"找回来"，memory 面向系统经历过的有用信息"留得住"。

### 为什么 Agent 需要 Memory

**1. 长任务不可能全靠上下文硬扛**
如果一个任务要持续几十步甚至几天，不可能把全部过程都一直塞进 prompt。需要把阶段性结论和关键事实沉淀成 memory 或持久化状态。

**2. 用户个体化体验需要持续**
例如用户偏好什么格式、常用什么术语、是否偏好简洁回答、所在行业、常用数据源，这些都不应该每次重新问。

**3. 经验会改变未来决策**
如果 agent 之前试过某个工具在某类任务上经常失败，或者某种搜索策略更有效，这些经验理应被保留下来，服务未来决策。

**4. 人类协同需要上下文延续**
人类接管、审批、编辑之后，系统需要记住人类修正过什么，否则下次还会犯同样错误。

### Memory 的经典分类

面试中最推荐的回答方式是分层，而不是一股脑说短期记忆、长期记忆。

**1. Working Memory（工作记忆）**
也叫运行时记忆。服务于当前任务的短时推理与执行，例如：当前已知事实、最近几步工具结果、当前计划和子任务状态、临时变量、中间摘要。通常生命周期较短，和 session state 有很强重合。

**2. Episodic Memory（情节记忆）**
记录系统曾经经历过的具体事件。例如：某次任务的步骤与结果、某次失败的原因、某次人工审批的修改意见、某个用户在某种任务上偏好的交付形式。情节记忆的特点是有时间、有场景、有具体经过。

**3. Semantic Memory（语义记忆）**
更像从多次经验中抽象出来的稳定知识。例如：用户偏好 Markdown 表格、某业务系统字段 status=3 表示已结案、某类任务常用三个数据源更可靠。它比情节记忆更抽象，也更适合作为长期可复用规则。

**4. Procedural Memory（程序性记忆）**
指"怎么做"的经验。例如：处理退款工单时优先检查订单状态再核对支付方式；调研类任务先搜英文资讯再补中文分析；修复代码 bug 时先跑最小复现。它接近策略模板或技能，通常非常有价值。

**5. User Memory（用户记忆）**
围绕某个用户的稳定偏好和背景。如岗位、团队、常用语言、输出格式偏好、时间区域、业务范围。这是企业产品中最常见的一类 memory，但也最容易涉及隐私与过时问题。

### Memory 的读写策略

**一）什么时候写入**

1. **任务完成后写总结** — 最安全的方式。任务结束后，把关键结论、有效策略、失败原因、人类修正点提炼成结构化记忆。
2. **在关键节点写 checkpoint（检查点）** — 长任务中可以在阶段结束时写入阶段性记忆，例如已验证来源、已完成子任务、关键证据。
3. **遇到高价值事件写 episodic memory** — 例如某个工具反复失败、某次人类明确修正偏好、某次关键审批被拒绝并附理由。
4. **不建议无脑全量写** — 把每轮对话和每个工具返回都直接塞进长期记忆，会迅速导致污染和检索噪声。

**二）什么时候读取**

1. **任务初始化时读用户相关记忆** — 例如用户偏好、所在项目、常用格式。
2. **规划前读任务经验** — 如果这是一个以前做过很多次的任务，可以在 planning 前先取回以往成功经验。
3. **执行受阻时读失败案例** — 当某工具失败，可以查历史上类似失败是如何处理的。
4. **最终输出前读交付偏好** — 确保回答格式与用户历史偏好一致。

> RAG 更像查资料，Memory 更像有过经历。

### Memory 存储设计要点

1. **不是所有 memory 都适合向量检索** — 用户偏好、角色信息、配额、当前审批状态更适合结构化存储。情节记忆和经验摘要则适合文本检索或向量检索。因此 memory 常常是混合存储，而不是单一向量库。

2. **记忆条目最好结构化** — 推荐包含：memory_id、类型（user/episodic/semantic/procedural）、作用域（session/user/project/org）、内容摘要、证据来源、置信度、更新时间、失效时间、可见权限、写入原因。

3. **需要记忆压缩与摘要** — 长历史不能全量保留为原文。应定期把冗长历史总结成高价值摘要，必要时保留指向原始日志的链接。

4. **需要评分与遗忘机制** — 并非越多越好。可以按最近使用、成功贡献、人工确认程度、置信度进行排序或衰减。

### Memory 的典型失败模式

**失败模式一：记忆污染**
系统把错误结论或幻觉写进长期记忆，下次反而更自信地重复错误。
解决：记忆写入前做验证；高价值记忆要求工具证据或人工确认。

**失败模式二：记忆过时**
用户偏好变了，业务规则更新了，但旧 memory 还在影响当前决策。
解决：时间戳、TTL（存活时间，Time To Live）、版本号、最后验证时间。

**失败模式三：记忆检索噪声大**
写了太多无关历史，检索时高价值内容淹没在噪声里。
解决：精写少写，分类型索引，按阶段选择性读取。

**失败模式四：记忆泄露权限**
把不该跨用户共享的 memory 当成公共知识召回。
解决：严格作用域、租户隔离、可见性策略。

**失败模式五：把日志当 memory**
日志适合审计，不适合直接进入模型上下文。需要先提炼成可消费的记忆。

**为什么不能把所有历史都直接存成长期记忆？**
因为会产生污染、噪声和过时问题。memory 需要提炼、验证、分层和遗忘机制。

**长期记忆和用户画像有什么区别？**
用户画像通常只是长期记忆里最稳定、最结构化的一部分，比如偏好、行业、岗位方向；长期记忆还包含经历、计划、失败经验、共享上下文和历史决策。所以 profile 是长期记忆的一部分，但长期记忆明显比 profile 更宽。

### session state vs working memory vs long-term memory vs cache

- **Session state**：当前任务的运行时状态（进度、临时变量、已完成步骤）
- **Working memory**：当前推理需要的短期信息（最近工具结果、中间摘要）
- **Long-term memory**：跨会话的持久化经验（用户偏好、成功/失败经验、历史决策）
- **Cache**：用于性能优化的临时副本，不是记忆

> 项目落地：当前流程状态放 session state，当前会话的临时结论放 working memory，用户偏好和跨会话信息放 long-term memory，企业制度和文档走 RAG。

**一句话总结 memory 设计原则：**
不是记得越多越好，而是要记真正有用、可验证、可控作用域、能在合适时机被正确取回的信息。

## 模型选型

**模型能力不是单一维度。**

### 1. 任务类型
- 问答/写作/总结 → 偏通用生成
- 代码修复/多步规划 → 偏 reasoning
- 分类/抽取/标签 → 偏结构化与低延迟
- 实时语音/高频 Agent → 偏低延迟
- 图文理解/文档分析 → 偏多模态
- 需要外部系统操作 → 偏工具使用可靠性
- 如果任务分类都不清楚，后面谈选型基本没有意义

### 2. 正确率要求
- 有些任务只要差不多能用（如草稿生成）
- 有些任务必须尽量别错（如合同审查摘要、报表分析、金融审批建议）
- 正确率要求越高，越值得上更强模型、更多验证链路或多模型复核

### 3. 延迟要求
- 面向用户对话、实时语音、IDE 辅助、在线客服，延迟非常敏感
- 如果业务要求首 token 极快，强模型质量再好也可能不适合
- 应明确区分离线批处理和在线交互，这是选型最基础的分层

### 4. 成本与吞吐
- 同样 100 万次请求，旗舰模型和轻量模型成本差一个数量级
- 并发还受上下文长度、KV Cache、缓存命中率和工具循环影响
- 真实系统常先用强模型打样，再 eval 驱动看能否下沉到更便宜模型

### 5. 上下文长度与管理能力
- 不能只看最大 context，还要看：
  - 长上下文利用率
  - prompt caching / context caching 支持
  - compaction / summary 能力
  - 多模态长上下文兼容性

### 6. 工具与结构化能力
- Agent 系统里，这一维经常比通用写作能力更重要
- 要看：function calling / tool use 是否成熟、structured outputs 是否严格、多工具并行/程序化工具能力、错误返回与 trace 可观测性、是否支持 web/file/code/computer use 等原生能力

### 7. 模型生命周期与发布阶段
- preview 模型可能能力很强，但行为变动快、稳定性波动大、参数建议变化快、可能被较快弃用或替换
- 选型不只是看今天能不能跑，也要看未来三个月稳不稳

> 真正系统设计时，这些维度往往相互牵制。例如：长上下文 ≠ 高利用率，强推理 ≠ 低延迟，高能力 ≠ 低成本。

### 实践方法（面试回答顺序）

**步骤 1：定义任务与指标**
- 先明确你要优化的是：准确率、格式合法率、延迟、成本、工具成功率、用户满意度
- **没有指标，就没有选型**

**步骤 2：用强模型建立上界（baseline，基准线）**
- 先用一个足够强的模型做 baseline，确定这个任务理论上能做多好

**步骤 3：做失败分类**
- 错误是因为模型不够强、证据不足、prompt 不好、工具不稳，还是输出解析崩？
- 不同根因，对应不同选型决策

**步骤 4：尝试向下路由**
- 把适合下沉的任务迁到更便宜更快的模型，观察质量损失是否可接受
- 这是最实用的降本路径

**步骤 5：补缓存、压缩和工作流优化**
- 很多时候不是非换模型不可，prompt caching、context engineering、structured outputs、tool schema 改善就能把模型表现拉上来
- 选型不应该掩盖系统工程问题

### 面试 Q&A

**Q1：怎么选模型？** 先看任务类型与验收指标，再看复杂度、延迟、成本、上下文、工具/结构化支持，最后通过样本评测做决策。

**Q2：为什么不直接用最强模型？** 因为不是所有请求都需要最高智能，最强模型往往更贵更慢；系统目标是综合最优而不是单点最强。

**Q3：什么时候上 reasoning 模型？** 多步推理、复杂代码、复杂约束一致性、多轮工具协作、高风险分析场景。

**Q4：什么时候适合小模型？** 分类、抽取、路由、改写、模板填充、高频轻量 Agent 子任务，以及对延迟/成本极敏感的场景。

**Q5：模型路由是什么？** 根据任务特征、复杂度、成本与 SLA，动态把请求分配给不同模型或模型组合的机制。

**Q6：preview 模型能上生产吗？** 可以试，但要谨慎。要准备行为波动监控、回退策略、版本对比和 deprecation 预案。

**Q7：如何做降本？** 先建立强模型 baseline，再通过 eval 发现可下沉任务，配合缓存、上下文压缩、结构化输出和更细粒度路由实现降本。

**Q8：长上下文模型就一定更适合长文档任务吗？** 不一定。还要看长上下文利用率、缓存能力、证据组织方式和实际任务结构。

**Q9：Agent 系统为什么常用多模型？** 因为规划、执行、抽取、总结等阶段需求不同，用单一模型串到底通常不经济。

**Q10：模型选型要不要考虑工具支持？** 必须考虑。Agent 系统里，function calling、structured outputs、native tools、错误处理能力往往和通用写作能力同等重要。

### 什么是 Agent

Agent 本质上是一个**围绕目标运行的系统**，而不只是一次输入输出的模型调用。

**四个核心特征**：
1. **围绕目标运行** — 给定目标持续推进直到结束，而不是给定输入生成输出
2. **在执行过程中做决策** — 每一步都可能根据环境反馈改变路线（搜索不够→继续检索，API 报错→换方式）
3. **借助外部能力推进任务** — 检索、数据库、网页、代码执行、文件系统、企业 API 等，与真实世界交互
4. **持续运行直到终止条件满足** — 存在运行循环：思考→选择动作→执行→观察→更新状态→再决策，直到完成/失败/超时/预算耗尽/人工中断

### 为什么 Agent 会火

Agent 解决的是**固定流程难以覆盖的开放式任务执行问题**。

**场景一：任务路径事先不确定**
- 例如"帮我调研最近一周行业新闻并写成分析"，先查哪几个网站、要不要改查询词、哪些信息要交叉验证，很难完全提前写死
- Agent 的价值在于让模型在运行过程中适配具体情况

**场景二：任务需要多轮环境交互**
- 电脑操作、浏览网页、调用多个系统 API、管理文件、运行代码等
- 需要**动作—观察—再动作的闭环**，而非一次文本生成

**场景三：任务规模超过单轮上下文**
- 复杂研究、长文档生成、跨多个数据源汇总、长期追踪式任务
- 需要规划、拆解、记忆、阶段性产物存储

**场景四：任务要求一定程度的自动化闭环**
- 从"给建议"升级为"帮你完成"
- 例：客服知识问答可能只需要 RAG；但要识别退款诉求→调订单→核验规则→发起审批→通知用户，就更接近 Agent

### Agent 适合与不适合的场景

**适合的场景：**
1. **路径不固定、需要探索** — 复杂研究、网页操作、多源信息搜集，很难穷举所有路径，agent 能减少硬编码
2. **明显依赖中间观察结果** — 工具返回什么直接影响下一步，如 API 字段缺失、数据库返回空，需要动态调整
3. **任务可拆阶段但需动态协调** — 先搜集→筛选→归纳→输出，每个阶段都可能因中间结果调整
4. **任务价值高于控制成本** — 单次成功收益高、人工成本高、流程变体多

**不适合的场景：**
1. **路径清晰且稳定** — 固定报表、标准审批、简单表单，用 workflow 更简单更稳更便宜
2. **错误代价极高又难以完全约束** — 高风险金融交易、关键生产控制，必须加强规则审批与沙箱
3. **数据与接口不稳定、但系统又不能容错** — 底层系统不稳定只会放大问题
4. **只是为了看起来高级** — 两段规则+一次检索能解决的，硬做成多 agent 缺乏成本意识

### 什么不算 Agent：
- **普通 chatbot**（聊天机器人） — 没有多步决策、没有工具使用、没有目标推进循环
- **固定流程 workflow** — 每一步预定义，没有动态决策能力
- **只有 function calling** — 没有多步循环和持续决策，更像 tool-augmented LLM
- **有记忆不等于 agent** — Memory 解决状态延续，Agent 解决目标推进

**与 chatbot 的区别**：Agent 强调目标驱动、多步决策、环境交互和状态持续
**与 workflow 的区别**：Agent 运行时有更高的决策自由度，不是所有步骤都预先写死

**工程组成**：目标 → 策略/规划器 → 工具/外部环境 → 状态/记忆 → 护栏/治理 → 校验器

**是否该用 Agent**：看任务路径是否不确定、是否需要多轮环境交互、引入自治后能否接受成本延迟和风险。

### Agent 面试 Q&A

**Q1：为什么 Agent 难做？**
难点不在于让模型想一步，而在于如何在复杂环境里持续正确做事。包括：**工具可靠性、上下文污染、权限控制、长任务恢复、终止条件、评测与回放**等。

**Q2：Agent 的核心组件有哪些？**
目标、策略/规划、工具、状态/记忆、控制约束、结果校验。这六块缺一不可，尤其在生产环境里。

**Q3：Agent 和多 agent 是一回事吗？**
不是。多 agent 只是 agent 系统的一种组织方式。很多场景单 agent 足够，而且更稳。多 agent 只有在上下文隔离、角色专门化、并行化或团队分工价值明显时才值得上。

**Q4：什么时候你不会建议上 Agent？**
任务路径清晰、规则强、结果必须稳定可控、错误代价高且可通过规则系统解决时，更倾向于 workflow 或纯程序逻辑。

**Q5：怎样判断一个需求是否应该 agent 化？**
看三点：任务路径是否难以预先编码；是否需要多轮环境交互；自动化收益是否足以覆盖引入自治后的复杂度和风险。

**Q6：为什么很多官方建议先从 workflow 开始？**
因为 workflow 更稳定、易测、可控、可回放、成本更可预测。只有在任务不确定性高到固定流程难以覆盖时，agent 的额外复杂度才值得引入。

### Agent vs Workflow 选择原则

> 当任务流程清晰、约束强、结果要求稳定时，**Workflow** 往往更合适；
>
> **Agent** 只有在路径不确定、需要动态规划或环境交互时，才真正发挥价值。

**Workflow 和 Agent 的本质区别**：
- Workflow 的核心是路径基本预定义，Agent 的核心是路径在运行时动态形成
- 这不是说 workflow 完全没有条件分支，也不是说 agent 没有任何规则约束
- 而是：**Workflow 的主要价值来自确定性编排；Agent 的主要价值来自面对不确定环境时的适应性**

**Workflow 算不算 Agent？** 稳妥回答：广义上有人把带一点动态性的工作流也叫 agentic workflow；但严格区分时，**workflow 侧重预先定义路径，agent 侧重运行时自主决策**。

### 为什么很多时候 Workflow 更好

1. **更可预测** — 路径和节点事先定义，容易知道请求经过哪些步骤、在哪些条件下终止
2. **更容易评测与回放** — 每个节点可设计独立测试集（检索召回率、抽取准确率等），agent 路径每次不同，debug 困难
3. **成本和延迟更可控** — workflow 调用次数固定；agent 有不确定的循环和试错，上界更高
4. **更易治理** — 权限、审计、审批、风控更容易做；agent 工具太多时治理难度显著上升
5. **更容易先上线** — 80% 场景用 workflow 解决，再逐步把不确定的 20% 场景 agent 化

### 什么时候 Agent 才真正有价值

为什么还需要 agent？答案是：**有些任务的路径没法被开发者提前穷举**。

1. **任务路径依赖环境反馈** — 例如浏览网页找信息，不知道页面结构是否变化、第一轮搜索是否足够。Agent 的强项是根据观察结果调整下一步
2. **任务需要探索** — 开放式研究、复杂文件理解、多源对照、电脑操作、跨系统流程。接下来怎么做本身就是求解的一部分。例如先收集候选方案→筛选→深入分析→补齐数据→输出结论，每个阶段都可能推翻前面的假设，用固定 workflow 会又长又脆
3. **任务步骤很多，且中间结果会改变计划** — 长上下文、分阶段交付、过程性校验、阶段性纠错，更适合 agent 系统
4. **任务规模和复杂性超过单轮输出**

### 混合架构

真正的生产系统里，workflow 和 agent 往往不是对立关系，而是**组合关系**。系统整体往往由 **deterministic workflow 包住若干 agentic 子任务**。

### 四种架构对比

| 方案 | 特点 | 适合 | 优点 | 缺点 |
|------|------|------|------|------|
| **纯 Workflow** | 固定节点，模型只是某些步骤的子能力 | 表单处理、意图路由、固定知识问答、结构化抽取、简单审批流 | 稳定、易测、成本可控 | 面对未知情况容易僵住 |
| **纯 Agent** | 模型主导大部分下一步决策，程序只给工具、状态和边界 | 开放式研究、电脑操作、复杂调试、跨系统探索 | 灵活，覆盖长尾场景能力强 | 不稳定、难治理、难预估成本 |
| **Workflow 包裹 Agent** | 外层确定、内层自主 | **企业落地中最常见** | 既保留灵活性，又保住治理与审计 | 设计边界时需要经验，否则两头都不纯 |
| **Agent 生成 Workflow** | agent 先规划，计划编译成 DAG，workflow runtime 执行 | 复杂任务拆解、并行执行、长期任务 | 兼顾规划灵活性与执行确定性 | 系统复杂度高，对状态和任务表示要求高 |

**企业案例**：员工发起"帮我整理竞品分析报告并发给老板"

```
外层 workflow（确定性）：
  鉴权 → 任务登记 → 配额检查 → 创建 trace → 路由 → 最终发送前人工审批
  ↑ 这些部分最好都固定、可控、可审计

中间 agent（灵活性）：
  自主搜索关键词 → 阅读多个来源 → 判断信息是否充分 → 必要时重规划 → 生成中间笔记与草稿
  ↑ 这一段需要更高的灵活性

收口 workflow（确定性）：
  校验格式 → 落库存档 → 调用邮件系统 → 记录结果与耗时 → 通知发起人
```

> 这类回答很加分，因为它显示你理解**自治应被包裹在确定性边界内**。

**六个判断维度**：路径可否预编码、结果容忍度、环境不确定性、可解释性、成本预算、迭代阶段。

### 常见反模式

1. **过度 agent 化** — 稳定流程硬让模型决定每一步，成本更高错误更多
2. **表面 workflow 实质乱跳** — 中间让模型自由决策，实际不可控
3. **Agent 没有外部约束** — 自由度给了但没有权限/审批/预算/停止条件
4. **把异常兜底也交给 agent** — 接口失败、权限不足应由程序层解决

> **工程上的奥卡姆剃刀**：能写死的先写死，不能写死的再交给模型。

### Agent 速记版

- Agent 的关键不是能聊天，而是能**围绕目标持续行动**
- Agent 的关键特征：**目标驱动、多步决策、环境交互、状态持续**
- 工具不是充分条件；**循环式决策才是核心**
- Workflow 强在稳定和可控，Agent 强在处理不确定路径
- 真正的 Agent 设计必须补齐：**权限、审批、日志、恢复、终止条件**
- 不是所有任务都值得 agent 化，过度 agent 化是常见反模式

### Agent vs Workflow 实战取舍

**核心原则**：先把确定的部分固化成 workflow，只把真正不确定、需要运行时决策的部分留给 agent。Workflow 和 Agent 不是低级 vs 高级的关系，而是**确定性编排 vs 自主性决策的取舍关系**。

**以企业知识助手为例**：

> 我不会把它简单做成纯 workflow 或纯 agent，而是会**拆层**。
>
> - **确定性步骤用 workflow**：鉴权、文档权限过滤、路由、日志、配额、结果落库、审批
> - **不确定部分用受约束的 agent**：开放式检索、跨文档对比、后续追问澄清、是否需要继续搜索
>
> 如果业务只是 FAQ、制度问答、标准文档查询，**workflow + RAG 就足够**；
>
> 只有当需求升级为**帮我跨系统搜集材料、整理观点、生成多阶段输出**时，agent 的价值才会显著。

## Agent 运行范式

### ReAct

**核心思想**：在每一步交替进行 Reasoning（推理）和 Acting（行动），并利用行动后的 Observation（观察）继续推进任务。

```
思考 → 决定动作 → 执行动作 → 获得观察 → 再思考
```

**为什么有效**：很多真实任务不是静态问答，而是互动求解（网页检索、数据库查询、代码调试、电脑操作）。ReAct 把模型从单轮生成器变成了交互式求解器。

**优点**：
- 简单直接，上手快
- 非常适合工具调用链路，易集成到 function calling
- 天然支持在每一步利用新观察修正方向
- 过程可 trace，便于调试 agent 决策

**缺点**：
- **局部贪心**：模型只根据当前观察做短视决策，没有全局规划
- **容易重复**：没设计好终止条件时，可能在两个工具间来回跳
- **上下文膨胀**：每步思考+动作+观察都写进上下文，长任务越来越重
- **难以并行**：默认串行循环，不适合天然可并行任务

**适合场景**：问题中等复杂但需多轮环境交互、不确定性来自环境反馈、需要快速构建 agent 原型、希望保留清晰过程 trace

### Plan-and-Execute

**核心思想**：先生成一个相对全局的计划，再按计划逐步执行，必要时重规划。

```
Planner 生成计划 → Executor 按步骤执行 → 反馈结果 → 需要时 replanning
```

**解决了 ReAct 的什么问题**：长任务中的全局性缺失。有了计划后，系统不容易每一步都做局部最优选择。

**优点**：更有全局视角、容易做任务拆解和职责分离、为并行化提供基础、便于插入检查点和审批点

**缺点**：计划可能一开始就错或很快过时、规划本身消耗 token 和延迟、执行器死板按计划跑会不灵活

**工程常见改造**：初始规划 + 分阶段重规划、计划只是软约束、关键路径固定细节步骤 agent 自主化

### CodeAct

**核心思想**：让模型通过**生成可执行代码**的方式表达动作，而不是只用自然语言描述下一步。

**为什么更强**：自然语言模糊且不可复用；代码更结构化、可执行、可复查、可复用。

**优势**：表达力强、代码可检查可测试、执行结果更客观错误易定位、适合与代码解释器结合

**风险**：代码执行带来安全问题、需要沙箱和资源限制、模型可能生成破坏性动作、调试链路更复杂

**与 ReAct / Plan-and-Execute 的关系**：CodeAct 是**动作表达形式的升级**，ReAct / Plan-and-Execute 是运行控制结构。可组合使用：ReAct + CodeAct、Plan-and-Execute + CodeAct。

### 三者对比

| 维度 | ReAct | Plan-and-Execute | CodeAct |
|------|-------|-----------------|---------|
| 控制结构 | 局部闭环、边做边想 | 全局先行、分步落地 | 不定义控制结构，动作通过代码表达 |
| 全局性 | 弱 | 强 | 取决于外层规划 |
| 灵活性 | 高 | 中 | 动作层灵活，执行环境约束强 |
| 并行潜力 | 低 | 高 | 代码层可支持较强并行 |
| 上下文负担 | 高 | 中 | 代码表示可减少冗余自然语言 |
| 安全性 | 防错误动作和循环 | 防计划失真 | 防代码越权和资源滥用 |

### 常见组合

1. **Plan-and-Execute + ReAct** — 先粗规划，子任务内用 ReAct
2. **ReAct + CodeAct** — 每轮观察后生成代码作为动作
3. **Planner 生成 DAG + 代码并行执行** — 适合并行子任务
4. **计划前置 + 关键动作审批** — 高风险动作走人工确认

### 如何选择（面试回答）

面试常见问题：面对一个复杂任务，你在 ReAct、Plan-and-Execute、CodeAct 里怎么选？

比较稳的答法是按任务特征分：

1. **任务短、探索性强、依赖实时反馈** → 优先 **ReAct**
   - 例如网页检索、简单工具链交互、即时问题求解

2. **任务长、步骤依赖明显、需要全局路径** → 优先 **Plan-and-Execute**
   - 例如多阶段研究、复杂报告生成、长流程业务执行

3. **动作本身需要精确操作、控制流或计算** → 优先引入 **CodeAct**
   - 例如数据处理、代码修复、自动化测试

4. **既长又复杂，还需要精确执行** → 不是单选，而是 **Plan-and-Execute 外挂 CodeAct**，并在子任务里保留 ReAct 式局部修正能力

### 高频追问：ReAct 为什么会循环？怎么治？

**原因**：
- 没有清晰的成功判定
- 工具结果不理想，模型不断尝试相似动作
- 历史上下文里没有有效总结，导致反复探索同一无效路径
- 缺乏失败记忆或去重机制
- 模型没有预算感知

**治法**：
1. 设最大步数、时间预算、token 预算
2. 对工具调用做去重检测
3. 每轮总结已经试过什么、为什么失败
4. 加 verifier（验证器），判断是不是应该停止或重规划
5. 对高风险动作要求人工确认
6. 用 planner 在较长任务前提供全局导向

### 三种范式的控制结构

- **ReAct**：思考 → 行动 → 观察（循环）
- **Plan-and-Execute**：生成计划 → 执行步骤 1 → N → 是否重规划（是→循环，否→结束）
- **CodeAct**：根据目标与上下文生成代码动作 → 在沙箱/运行时执行 → 获得执行结果/报错

### 场景题：让 Agent 修复线上报错，怎么设计？

**参考回答**：
- 如果只是根据日志做一次性分析，先用 **ReAct**，因为它需要边看错误、边查代码、边搜索信息
- 如果任务升级成定位问题→修改代码→运行测试→生成修复说明→提交 PR 草稿，则用 **Plan-and-Execute + CodeAct**
  - Planner 把任务拆成：日志定位 → 相关文件检索 → 修复候选生成 → 测试验证 → PR 总结
  - 执行器通过代码与命令行动作在沙箱里完成每一步
- 限制代码执行权限，高风险动作（git push、修改生产配置）要求人工确认
- 为每阶段记录结构化产物和回滚点

**更优解：Workflow 编排 + Agent 处理不确定环节**
按"能写死的先写死"原则，不应该把全部步骤交给 planner，而是把确定性操作写成 workflow 节点：

```
Workflow 层:
  1. 日志分析定位根因     → Agent（不确定，需要模型判断）
  2. git checkout -b fix  → 写死（确定性操作）
  3. 生成修复代码         → Agent（不确定）
  4. 运行测试套件         → 写死（确定性操作）
     ├─ 通过 → git commit + PR  → 写死
     └─ 失败 → 回到步骤 3，Agent 根据测试输出修正
```

这样可以减少不必要的模型调用，降低成本和延迟，也更可控。

### 高频面试题

**题 1：Plan-and-Execute 一定更强吗？**
不一定。它也有规划成本，而且计划可能很快过时。若任务较短或环境变化快，过度规划反而浪费。

**题 2：如果任务既长又复杂，你会怎么设计？**
通常会采用混合模式：先规划，再执行；执行中保留 ReAct 式局部修正；对复杂计算或系统操作用 CodeAct；对高风险动作做人类审批。

### Task Decomposition（任务拆解）与 Planning（规划）

任务拆解与 planning 不是让模型先列个步骤，而是把**开放目标转成可执行、可管理、可恢复的任务结构**。

#### 为什么需要任务拆解

大模型面对复杂开放式目标时，不给结构容易出现三种问题：
- **过粗**：一步跨太大，遗漏关键约束
- **过细**：在低价值小动作上浪费步数
- **迷失**：没有阶段概念，做着做着就跑偏

核心价值：把开放目标转成更稳定的执行单元。

#### 拆解粒度

**拆得太粗**：子任务仍然很大（如"去研究竞品"），planning 形同虚设，无法明确成功标准、无法并行、无法精确回退。

**拆得太细**：拆到"打开网页→滚动页面"级别，环境稍有变化计划就过时。token 开销大、高层目标被埋没、系统变脚本录制器。

**合适粒度**的三个条件：
1. 有清晰的输入和输出
2. 有相对稳定的成功判定
3. 在必要时可由不同执行器完成

#### Planning 的表示形式（不止步骤列表）

1. **线性步骤列表** — 最简单，适合串行任务。表达力有限，不适合并行和分支
2. **树状计划** — 目标到子目标的层级分解。适合分层规划，但需额外机制转换运行顺序
3. **DAG（有向无环图）** — 依赖+并行混合时最合适。便于 runtime 调度、并行执行和失败重试
4. **状态空间/搜索树** — 如 Tree of Thoughts，探索多个候选路径、打分剪枝。适合复杂推理，注意成本
5. **程序化计划** — 结构化 schema、JSON、DSL，直接编译成执行图。最适合和 agent runtime 集成

> 计划的价值不在于写下来，而在于能否被执行器、监控器和治理层真正消费。

#### Planning 的核心流程（闭环）

1. **目标澄清** — 明确成功标准、范围、限制条件、预算时间约束。很多 planning 失败根源是目标本身不够清楚
2. **初始分解** — 拆成候选子任务，明确输入依赖、预期输出、成功判定、工具类型、可否并行
3. **计划检查** — 执行前静态检查：遗漏约束？循环依赖？不可执行？越权？成本过高？
4. **执行与观测** — 按计划推进，每步产生结果、日志和状态更新。planning 持续提供"为什么做这一步"的上层语义
5. **进度评估与 replanning** — 某步失败、环境变化、信息不足、目标变化时决定是否重规划。规划一次跑到底的系统不够鲁棒
6. **收口与复盘** — 存档过程产物，提炼可复用经验（默认模板、失败模式、工具适配建议）

#### 静态 vs 动态规划

- **静态规划**：开始时生成完整计划再执行。全局性强、便于审批，但对环境变化适应差
- **动态规划**：执行中不断调整，或先粗规划、局部细化。灵活但系统复杂度高
- **工程折中**：先生成高层计划 → 每个阶段开始前局部细化 → 关键节点决定是否 replanning。兼顾全局性与灵活性，最实用

#### 面试高频点：Planning 和 Prompt Engineering 有什么关系

有些候选人会把 planning 讲成写个 prompt，让模型先列计划。这是不够的。更好的回答是：

1. Prompt engineering 可以影响 planning 的质量
2. 但 planning 本身是**系统能力**，不只是提示词技巧
3. 真正有效的 planning 需要配合结构化输出、状态管理、执行器、验证器、回滚和 replanning 机制
4. 否则模型只是说了一个计划，系统却未必真的按这个计划可控地运行

### Planning 的典型失败模式

**失败模式一：计划很好看，但不可执行**

**失败模式二：子任务之间接口不清**
例如上游输出不是下游真正需要的格式，导致中间产物无法复用。

**失败模式三：过度规划**
对简单任务先规划一大堆，结果规划成本比任务本身还高。

**失败模式四：不重规划**
明明环境已经变化，系统仍死守旧计划。

**失败模式五：把计划当真理**
计划应当是指导而非神谕。现实世界的 agent 需要留出校正空间。

### 高频面试题与参考回答

**题 1：为什么 Agent 需要 task decomposition（任务拆解）？**
因为复杂目标通常不能直接执行。拆解能降低单步复杂度、暴露依赖关系、支持并行与阶段性交付，并提升可控性与可评测性。

**题 2：任务拆解时最重要的是什么？**
不是拆得越多越好，而是拆到合适粒度。每个子任务都应有清晰输入输出、成功标准和相对稳定的执行边界。

**题 3：Planning 和 Workflow 是一回事吗？**
不是。Planning 是把目标转成执行结构，Workflow 是确定性编排。Planning 可以生成 workflow，也可以为 agent 提供运行指导。

**题 4：什么时候需要 replanning？**
当关键假设被新观察推翻、执行失败无法局部修复、目标变化、预算紧张或发现更优路径时。

**题 5：计划应该用自然语言还是结构化表示？**
工程上更推荐结构化表示，至少要包含步骤、依赖、输入输出、工具、成功判定等字段，便于执行与治理。

**题 6：为什么不能把计划拆得太细？**
因为过细计划会脆弱、昂贵、难维护，而且容易把高层目标淹没在微动作里。

**题 7：为什么也不能拆得太粗？**
因为执行器仍然会重新面对一个复杂开放问题，计划失去作用，也难以评测和回滚。

**题 8：计划失败最常见的原因是什么？**
目标不清、步骤不可执行、依赖关系不明、没有完成条件、环境变化后不重规划。

**题 9：如何让 planning 真正提高系统质量？**
结构化计划、计划检查、阶段产物持久化、明确完成条件、支持重规划和部分完成交付。

**题 10：一句话总结你对 planning 的理解？**
Planning 不是让模型写一个好看的步骤列表，而是把开放任务转成可执行、可监控、可恢复的任务结构。

## RAG（检索增强生成）

### 上下文窗口越大越好？
- **不一定**
- 更大窗口带来：
  - **更高成本** — 每请求处理更多 token
  - **更高首 token 延迟** — 处理长上下文需要更多预填充时间
  - **更大缓存压力** — KV Cache占用随上下文线性增长
  - **利用率未必线性提升** — 模型对长上下文的关注力并不均匀，中间信息容易被忽略（"lost in the middle"）
- 关键：**够用就好**，根据实际场景选择合适窗口大小

### 长上下文 ≠ 不需要 RAG
- **长上下文并不保证高效利用** — 关键证据可能被埋没（lost in the middle）
- **RAG 的优势**：
  - **减少噪声** — 只注入最相关的片段，而非整个知识库
  - **提升证据密度** — 注意力集中在高质量信息上
  - **降低成本** — 处理更少 token意味着更低延迟和费用
- **一句话总结**：长上下文解决的是**放得下**，而**检索、重排、压缩、缓存**解决的是**用得好、用得省、用得快**

### Chunking：切块到底在切什么

切块的**本质**不是把长文本裁成固定长度，而是为检索系统定义一个**可被索引、可被召回、可被引用、可被拼装的最小语义单元**。一个好的 chunk 至少要满足四件事：包含相对完整的语义；不要引入太多无关内容；能挂载足够多的元数据；长度要适合后续 embedding、召回和上下文预算。

**常见切块策略分四类：**

1. **固定窗口切块** — 按 token 或字符数切成 256/512/800/1024 大小并加 overlap。简单、吞吐高、可控性强，是许多生产系统的起点。OpenAI file search 默认 chunk size 为 800 tokens、overlap 为 400 tokens。

2. **基于结构的切块** — 按标题、章节、段落、表格、代码块、问答对、页面块来切，比纯固定窗口更能保留语义边界。

3. **语义切块** — 通过 embedding 相似度、语义边界检测或句间断点识别来决定分块边界。

4. **层级切块** — 同时保留 coarse chunk 和 fine chunk，用于先粗定位、再细取证。

**切块不是越细越好**：chunk 太小会导致上下文不完整（制度文档、表格说明、长代码函数、合同条款）；chunk 太大又会把噪声带进来，导致向量平均化、召回不准。

**Overlap（重叠窗口）、标题增强与元数据为什么重要**：Overlap（重叠窗口）防止重要信息被切在边界上，但过大意味着索引膨胀、重复召回、重排浪费。更重要的是标题增强与元数据保留——把章节标题、子标题、文档标题、目录、更新时间拼到 chunk 前缀里，或作为 metadata 参与过滤和排序，效果往往不亚于更换 embedding 模型。

> 切块时保留结构信息、来源信息和权限信息，对企业知识库来说，和能否召回同样重要。

### RAG 高频面试题

**Q1：什么是 RAG？为什么比直接让模型回答更适合企业知识问答？**
RAG 是 Retrieval-Augmented Generation，先从外部知识源检索证据，再基于证据生成答案。它比直接让模型回答更适合企业场景的原因有三：知识可更新、私域知识可接入、答案可追溯。但 RAG 不能天然保证正确，还需要评测、权限和拒答机制。

**Q2：RAG 和微调有什么区别？什么时候用哪个？**
微调主要改模型行为和输出风格，RAG 主要补外部知识。知识频繁变化选 RAG，输出格式/行业表达不稳可用微调，生产里常常二者结合。最稳妥的表述：RAG 补知识，微调改行为。

**Q3：RAG 和长上下文模型谁更好？**
不能绝对比较。长上下文解决的是容量问题，RAG 解决的是从大规模候选中选出真正该看的那一小部分。在知识库较小、权限简单、整库放进上下文更省事时，直接 long-context 往往更简单；在知识源很大、更新频繁、权限复杂、成本敏感时，RAG 更合适。更成熟的工程说法不是"谁替代谁"，而是路由：小库直接 long-context，大库走 RAG，复杂任务走"检索 + 长上下文综合推理"。

**Q4：RAG 的核心链路是什么？**
面试里最好把 RAG 拆成离线链路和在线链路。

离线链路：数据接入 → 解析与标准化 → 切块 → 索引构建 → metadata 和 ACL（访问控制列表）绑定。
在线链路：查询理解 → 召回 → 重排 → 上下文组装 → 生成答案 → 返回引用 → trace 和回放。

离线链路决定知识能否被正确表征，在线链路决定证据能否被正确使用。

**为什么说 RAG 不是一个"向量库项目"？**
因为向量索引只是召回层的一种实现，真实生产系统还包含：
1. 文档解析与清洗
2. 切块和 metadata 设计
3. dense（稠密）/ sparse（稀疏）/ hybrid（混合）召回
4. rerank 和上下文组装
5. 引用、拒答、权限、审计、评测和回归

所以更准确的说法是：RAG 是知识访问、证据约束、系统治理和成本控制的组合方案，不是单点向量检索技巧。

**RAG 的核心指标有哪些？**
一定不要只说"看回答准不准"。更完整的讲法是拆成三层：

1. **检索指标**：Recall@k、MRR、nDCG、Hit Rate
   - **Recall@k（前 k 召回率）**：前 k 个结果中命中的相关文档数 ÷ 全部相关文档总数，衡量有没有把正确答案找回来
   - **MRR（Mean Reciprocal Rank，平均倒数排名）**：第一个正确答案出现在结果列表第几位，取倒数后平均。适合只有一个正确答案的场景
   - **nDCG（Normalized Discounted Cumulative Gain，归一化折损累计增益）**：既看相关文档有没有被召回，也看排在第几位，排得越靠前得分越高，对排名做对数折损。适合多级相关性评分
   - **Hit Rate（命中率）**：前 k 个结果中是否至少包含一个相关文档。比 Recall@k 更宽容，适合"只要找到一条就行"的场景
2. **生成指标**：正确率、faithfulness（事实忠实度，回答是否忠于给定证据）、citation accuracy（引用准确率，引用的内容是否真的支持对应结论）、拒答准确率、hallucination rate（幻觉率，模型无证据编造的比例）
3. **系统指标**：P95 延迟、单问成本、索引更新时间、权限错误率、失败重试率

高分表达是：RAG 的问题定位必须按链路拆指标，否则你根本不知道问题在召回、排序还是生成。

**为什么很多 RAG demo 看起来能跑，线上却不好用？**
因为 demo 只验证单轮 happy path（理想路径），而线上会遇到脏文档、表格、代码块、权限、长尾术语、版本冲突、查询分布漂移和性能约束。很多 demo 连引用、回放、ACL、增量更新和回归测试都没有，所以只能证明"能跑"，不能证明"能上线"。

面试里可以用一句话收尾：**demo 验证的是可行性，生产系统考验的是稳定性、可追溯性和治理能力。**

**chunk size 应该怎么定？**
没有固定值，从四个维度来定：

1. **按内容类型定范围**：知识百科/文档问答用 256-512 tokens（一段话一个概念）；代码/技术文档用 512-1024 tokens（一个函数或方法块）；长文报告/合同用 1024-2048 tokens（保持章节语义完整）。
2. **核心权衡**：块太小 → 上下文碎片化，模型看不到完整语义；块太大 → 噪声多、检索精度下降。
3. **经验法则**：从 512 tokens 起步，配上 10-15% overlap，先用 eval 测 Recall@k。召回缺关键信息则增大，召回了无关内容则减小。
4. **多策略混合**：小块用于精确召回，关联的大块（parent chunk）作为补充上下文送给模型。

**一个 RAG 系统为什么会答错？**
至少四类原因：解析切块错、召回错、重排/上下文组装错、生成越界。面试里要体现你会分层定位，不要笼统说模型幻觉。

**什么时候不应该上 RAG？**
当知识库很小、权限简单、整库放进上下文更省事时；或者任务本质不是查知识而是改行为时。Anthropic 甚至建议小于约 20 万 token 的知识库可以直接塞进上下文。

**如何理解 parametric memory 和 non-parametric memory？**
参数记忆是模型训练进参数里的知识，更新成本高、不可追溯；非参数记忆是外部知识索引，可热更新、可审计。RAG 的核心就是把二者结合。

### Chunking-Embedding-Rerank 面试题

**Q5：为什么说 chunking 决定了检索系统的上限？**
首阶段召回只能在已有 chunk 上选，如果关键语义被拆散，后续 embedding 和 rerank 再强也只能在坏候选里挑。

**Q6：overlap 有什么作用，为什么不能太大？**
减少边界截断损失；太大导致索引膨胀、重复召回、rerank 浪费。

**Q7：embedding 模型怎么选？**
看领域、语言、查询分布、维度、延迟、成本和向量库支持，不要只看榜单。

**Q8：为什么很多系统要加 rerank？**
首阶段召回相关性判断不够精细，rerank 用更细粒度模型重新排序，提升 top-k 质量。

### Agentic RAG / GraphRAG

**Agentic RAG：** 当问题需要多步取证、跨工具或跨知识源、先查 A 再查 B 时，单次 top-k 检索不够。检索策略应随中间结果变化：先判断问题复杂度 → 对复杂问题做 decomposition → 针对不同子问题路由不同知识源 → 证据不足则改 query 或继续追检。

**GraphRAG：** 将文档构建为知识图谱（实体 + 关系），支持多跳推理、全局摘要和关系型问答。适合需要链路推理的场景，但构建和维护成本高。

### RAG 常见失败模式

| 失败模式 | 说明 | 解决方向 |
|---------|------|---------|
| 文档解析/切块失败 | 表格被读坏、标题丢失、代码函数被切断 | 结构化解析，分类型处理 |
| 召回失败 | 关键证据不在 top-k 中 | hybrid retrieval + 调 chunk + rerank |
| query 表达不匹配 | 用户问题与文档 chunk 分布不一致 | query rewrite / expansion |
| 权限过滤错误 | 越权数据被召回或应该召回的没召到 | ACL（访问控制列表）从 ingest 阶段下沉 |
| 上下文组装失败 | 太多无关 chunk 挤占上下文 | 精排 + 压缩 + 引用约束 |
| 生成越界 | 模型不基于证据作答 | 强制引用 + 拒答机制 |

## Multi-Agent（多智能体）

### 什么时候多 Agent 才值得做

多 agent 真正有价值，一般出现在下面几类情况：

1. **上下文太杂，需要隔离** — 单 agent 面对复杂任务时往往需同时记住任务目标、工具说明、用户偏好、多个中间结果、多个候选路径，上下文一旦拥挤，决策质量就会下降。把不同职责拆给不同 agent，可以降低单个 agent 的上下文负担。

2. **任务天然存在专业化角色** — 角色边界本身就很清晰时，不同子任务可采用不同模型、不同工具、不同 prompt 和不同校验方式。

3. **任务可并行** — 如果任务中有多个独立信息源或子任务，拆成多个 sub-agent 并发执行可能明显降低时延。

4. **团队协作与模块化开发需求强** — 在大型工程团队里，多 agent 也有组织上的好处：不同模块可独立开发、测试和维护。

5. **需要不同能力边界和权限边界** — 通过 agent 分工，有时比在一个大 agent 内部做复杂权限表更清楚。

### Multi-Agent 的几种典型架构

**1. Manager-Worker（主管—执行者）**
最经典。一个 manager 负责理解总目标、拆解任务、分派给多个 worker，并聚合结果。
- 优点：管理结构清晰，容易加入高层 planning，便于统一收口和最终校验
- 缺点：manager 容易成为瓶颈；manager 拆错则整个系统偏；worker 之间协作弱，容易信息孤岛
- 适用：总目标明确、子任务可清晰切分、需要中心化控制

**2. Handoff（接力式切换）**
当前 agent 判断应由谁负责后，把控制权交给另一个 agent。更像客服转接或角色接力。
- 优点：符合某些业务流程；每次只有一个当前主 agent，状态更容易理解；适合阶段式任务
- 缺点：接力边界设计不好会频繁来回切换；上下文传递质量决定效果；容易出现没人总览全局
- 适用：角色阶段性明显，如售前→实施→售后

**3. Router + Specialist**
先有一个路由器，根据任务类型把请求送到最合适的专家 agent。
- 优点：结构简单，易于扩展专家 agent，适合任务类型分布清晰的场景
- 缺点：主要解决路由，不一定解决复杂协作；路由错了整个路径就偏
- 适用：多模态、跨领域、多业务线系统

**4. Blackboard / Shared Workspace**
多个 agent 围绕一个共享任务板、共享记忆区或共享文档协作。
- 优点：有利于并行协作，中间产物可复用，减少点对点通信
- 缺点：共享区容易污染；谁负责最终裁决不清楚时会混乱；对状态管理要求高

**5. Hierarchical Planning + Subagents**
高层 planner 只做任务图和资源分配，具体子任务交给 specialized subagents。常见于复杂研究或代码任务。

### Multi-Agent 的核心设计点

**1. 拆分依据必须明确**
常见拆分维度：按任务阶段拆、按专业能力拆、按数据源拆、按风险级别拆、按权限边界拆。如果拆分依据不清，多 agent 只会变成多几个 prompt。

**2. Agent 间传什么，不传什么**
这是最关键的问题之一。传全部上下文则失去拆分意义；传太少则下游决策失真。稳妥做法：传结构化任务说明、必要约束、关键证据摘要和明确的 expected output，而不是原样转发全部历史。

**3. 共享状态如何管理**
系统需要明确：哪些状态是全局共享、哪些只属于某个 agent、谁可以写共享 memory、谁拥有最终裁决权。

**4. 结果如何归并**
多个 worker 返回的结果可能冲突、重复、质量不齐。需要 aggregator（聚合器）或 reviewer 做统一整合、冲突解决和证据校验。

**5. 出错时怎么定位**
错误传递链：A 拆错任务 → B 根据错任务输出 → C 据此做总结，最后看起来像 C 答错了，但真正问题在 A。trace 必须记录谁分派了什么、传了哪些上下文、下游基于什么证据做决定。

### 为什么 Multi-Agent 容易失败

1. **通信损失** — Agent 之间传递的信息往往是压缩后的，压缩一定有损。如果任务说明不完整，下游 agent 可能在错误前提上行动。

2. **角色漂移** — 本该只做事实收集的 agent 开始擅自做结论；reviewer 不只审核还重新发明方案。职责不清导致系统行为失控。

3. **错误放大** — 单 agent 出错通常是一处错误；多 agent 出错可能层层放大。上游错误被下游当成事实时问题更严重。

4. **成本高** — 每个 agent 都有自己的系统提示词、上下文和输出，通信越多成本越高。

5. **调试困难** — 要 debug 的不再是一个模型调用，而是一个网络。问题可能来自拆分、传递、归并、顺序、权限、上下文摘要等多个环节。

6. **收益不一定显著** — 在很多任务上，一个更强的单 agent 外加好的工具、planning 和状态管理，效果可能比多 agent 更好、更便宜、更稳。多 agent 必须证明自己的边际收益。

### 如何判断是否该上 Multi-Agent

- **维度一**：单 agent 是否已出现明显瓶颈 — 上下文过载？角色冲突？难以并行？prompt 越写越长、越不稳定？
- **维度二**：任务是否存在自然分工 — 不同步骤是否需要不同能力？是否可以清晰定义输入输出？
- **维度三**：拆分后的通信成本是否可接受 — 子任务结果能否结构化传递？是否会产生大量往返？
- **维度四**：是否有明确的治理需求 — 不同 agent 是否对应不同权限或审批要求？
- **维度五**：收益是否大于复杂度 — 成功率是否显著提升？时延是否下降？可维护性是否变好？

### 高频面试题

**Q1：为什么不直接用一个更强的单 Agent？**
因为多 agent 会引入通信成本、调试难度和额外上下文开销。只有当单 agent 已在上下文管理、角色分工或并行需求上出现明显瓶颈时，多 agent 才值得引入。

**Q2：多 Agent 的主要收益是什么？**
上下文隔离、专业化角色分工、并行执行、模块化开发，以及在某些任务上的更清晰责任边界。

**Q3：Manager-Worker 和 Handoff 有什么区别？**
Manager-Worker 是中心化分派和汇总；Handoff 是控制权在不同 agent 间接力式转移，通常每个阶段只有一个主导 agent。

**Q4：多 Agent 最难的工程问题是什么？**
上下文如何传递、共享状态如何管理、结果如何归并，以及错误发生时如何定位是哪一层出了问题。

**Q5：什么时候多 Agent 反而会更差？**
任务本身不复杂、通信成本高、上下文摘要有损、角色边界不清，或者一个强单 agent 已能胜任时。

**Q6：如何设计 agent 之间的通信？**
尽量结构化传递任务说明、关键约束和证据摘要，而不是原样搬运全部历史。让输入输出协议清晰、可验证、可追踪。

**Q7：多 Agent 如何做权限控制？**
按角色划分工具与权限，不同 agent 只暴露最小必要能力；高风险 agent 或动作要额外审批和审计。

**Q8：多 Agent 如何做结果归并？**
需要用 aggregator（聚合器）或 reviewer 对冲突、重复和证据质量做统一处理，而不是简单拼接。

**Q9：怎样判断是否应该上多 Agent？**
先看单 agent 是否已遇到上下文过载、专业化冲突或并行瓶颈；再看拆分后通信成本和治理复杂度是否可接受。

**一句话总结你对 Multi-Agent 的态度？**
它是解决复杂性的一种架构手段，但只有在分工收益明显大于通信与治理成本时才值得做。

### 多 Agent ≠ 人人都是完整 Agent

真实系统里，很多所谓 multi-agent 其实是不同等级的 agentic component：
- 有的是完整自治 agent
- 有的是带工具的小专家节点
- 有的是只做分类或评审的 evaluator
- 有的是 skill 或 subroutine

多 agent 系统不一定由多个平权智能体构成，很多时候是中心化 runtime 管理若干不同自治程度的子单元。

### Agent Runtime（运行时）设计

Runtime（运行时）是 agent 运行的底层基础设施，至少包含：

| 能力 | 说明 |
|------|------|
| Function Calling 协议 | 模型如何表达工具调用意图 |
| MCP 接入 | 标准化的工具/能力接入协议 |
| Session State 管理 | 当前任务运行时状态 |
| Durable Execution（持久化执行） | 长任务可中断恢复 |
| 任务调度 | 重试、幂等、回滚 |
| 异步/长任务/流式 | 非同步执行支持 |
| Skills 插件化 | 能力封装与动态加载 |
| 权限/审计 | 多租户隔离与安全治理 |

**Agent Runtime 的核心职责：** 接收模型输出 → 校验与路由 → 执行工具/动作 → 回传结果 → 管理状态 → 处理异常 → 支持恢复与审计。

**Runtime 如何做到 Durable Execution（持久化执行）？** 通过持久化状态机 + 事件日志 + 幂等键，在进程重启或中断后从 checkpoint（检查点）继续，而不是从头重跑。

### Skills、插件化与能力封装

**Tool → Skill → MCP Server → A2A Agent 的区别：**

| 层级 | 定义 | 自治程度 |
|------|------|---------|
| **Tool** | 最细粒度的原子动作 | 无 |
| **Skill** | 一组动作 + 指令 + 资源 + 经验的封装方法包 | 低 |
| **MCP Server** | 通过标准协议暴露能力（tools/resources/prompts） | 中 |
| **A2A Agent** | 完整自治体，有身份、任务状态和协作语义 | 高 |

Skill 的核心价值：边界清晰、方法可复用、依赖显式、输出标准明确、可评测、可版本化。常见误区：Skill 不是 prompt 模板；Skill 不是工具替代品；Skill 越大越好是错的。

#### Human-in-the-Loop（HITL，人工介入）

HITL（Human-in-the-Loop，人工介入）不是最后点一下确认，而是在 agent 运行过程中，让人类在关键节点对计划、动作、状态或结果进行审批、编辑、驳回、澄清或接管。

**两个关键点：** 关键节点（不是每一步都让人看）和介入方式（审批/编辑/驳回/接管/纠错）。

**典型介入点：** 规划完成后、高风险动作前（发邮件、批量更新、执行脚本）、工具结果异常时、最终输出前。

**工程关键：** HITL 本质引入状态 `WAITING_HUMAN`，需要状态机支持挂起与恢复。没有状态机，HITL 就是前端弹个框。

## 生产优化

### 模型参数与生产实践
- **参数不是越大越好**
- 生产上追求的是 **质量 / 成本 / 延迟** 的最优平衡，而不是盲目追大
- 小模型在特定场景下可能更高效、更经济

### 成本误区：只看 token费用，不看并发与显存
- **很多人只算 token 单价（$/M tokens），但线上服务真正瓶颈往往是**：
  - **KV Cache爆炸** — 每处理一个长上下文请求，KV Cache占用的显存随 batch size（批次大小）× 上下文长度线性增长；高并发时显存迅速耗尽
  - **TTFT（Time To First Token，首词元延迟）拖慢** — 长上下文的 prefill（预填充）阶段计算量大，首 token 延迟飙升，影响用户体验
  - **并发能力受限** — 显存被 KV Cache吃满后，能同时服务的请求数骤降
- **正确评估成本**：不能只看 token 单价，要综合考虑**吞吐量（TPS，Throughput Per Second）、并发数、显存占用、TTFT 延迟**
- **优化方向**：量化推理、Flash Attention（闪存注意力）、PageAttention（分页注意力，vLLM）、prefix caching（前缀缓存）、continuous batching（连续批处理）
- **KV Cache** 解决单次生成的重复计算，**Prompt / Context Caching（提示缓存 / 上下文缓存）** 解决跨请求共享前缀的重复 prefill（预填充） — 它们是长上下文时代的核心工程武器

### 提高缓存命中率（Prompt Cache，提示缓存）
- **稳定 system prompt（系统提示）和工具 schema（模式）** — 固定部分不变是缓存命中的基础
- **减少无意义动态字段** — 如时间戳、随机 ID 等每次不同的字段会打乱缓存
- **固定前缀与用户动态输入分离** — system prompt（系统提示）+ 工具定义作为固定前缀缓存，单独的 user message（用户消息）放动态内容
- 典型做法：多轮对话中**复用 system + tools 的 KV Cache**，只重新计算用户输入部分

### 生产实践案例

> "我们最初用单一的大模型处理所有请求，效果还可以，但成本和延迟都偏高。后来把链路拆开做路由：复杂规划和最终审定仍交给强模型；文档抽取、证据整理、分类路由下沉到轻量模型；同时对 system prompt 和工具 schema 做稳定前缀，提升缓存命中率。选型时我们不是看排行榜，而是根据任务完成率、格式合法率、平均延迟和单位请求成本做评测。最终整体质量基本持平，但成本显著下降，在线时延也更稳定。"

**关键做法**：
1. **链路拆分** — 强模型做复杂推理，轻量模型做抽取/分类/路由
2. **缓存优化** — system prompt 和工具 schema 稳定前缀化，提升缓存命中率
3. **评测驱动选型** — 不只看排行榜，看任务完成率、格式合法率、延迟、单位成本

### 高频面试速记

**Q：从零设计一个 Agent 系统，你会怎么做？**
1. 确定边界：是 workflow 还是 agent
2. 选模型：按任务类型 + 延迟 + 成本
3. 设计 tool schema：最小权限、结构化参数
4. 搭 Runtime：状态机 + 工具执行器 + 异常处理
5. 补治理：HITL + 审计 + 权限 + trace
6. 加记忆：session state + working memory + long-term memory
7. 评测：按链路拆指标，不只看最终答对没

**Q：RAG 线上不好用，排查路径是什么？**
文档解析坏 → 切块边界错 → 召回不全 → query 表达差 → rerank 没加 → 上下文组装乱 → 模型不按证据答。按链路一层层查，不笼统说"模型幻觉"。

**Q：MCP 和 Function Calling 的区别？**
FC 是模型表达调用意图的协议，MCP 是工具/能力如何被标准化接入的协议。FC 解决"怎么说"，MCP 解决"怎么接"。
